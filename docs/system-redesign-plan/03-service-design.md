# ZKBot Service Design

Per-service responsibilities, startup contracts, and domain reuse mapping.

## 1. Control Plane

### 1.1 ZK Pilot Service

Language: Python (FastAPI for REST; grpcio for internal gRPC)

Responsibilities:
- operator REST API: account management, config CRUD, strategy lifecycle orchestration
- strategy execution registration: `POST /v1/strat/strategy-inst` (engine registers on startup, deregisters on shutdown); validates and leases `instance_id` for Snowflake order-ID uniqueness
- internal gRPC query endpoints: topology, risk state, execution state
- NATS request/reply bootstrap authorization for instance registration (token validation + grant issuance)
- MDGW subscription management: admin UI writes `cfg.mdgw_subscription` table; publishes `zk.control.mdgw.<venue>.reload` on change
- subscription and scheduled job management (refdata sync, recon jobs)
- instrument refdata serve (replaces ODS `query_instrument_refdata` and `query_account_trading_config`)
- watch NATS KV for live topology

Bootstrap subjects (NATS request/reply):
- `zk.bootstrap.register`
- `zk.bootstrap.reissue` (optional)
- `zk.bootstrap.deregister` (optional)
- `zk.bootstrap.sessions.query` (optional admin query)

Bootstrap flow:
1. instance connects to NATS using `ZK_NATS_URL`
2. instance sends request to `zk.bootstrap.register` with token + runtime metadata
3. Pilot validates token/topology/duplicate policy
4. Pilot returns scoped registration grant (`kv_key`, `lock_key`, `owner_session_id`, TTL, scoped credential)
5. instance registers and heartbeats directly to KV (no Pilot in steady-state heartbeat path)

Config sources:
- PostgreSQL `cfg.*` tables: accounts, OMS instances, gateway instances, bindings, refdata
- NATS KV `zk.svc.registry.v1`: live service registrations (read-only watch)

Reference: `services/zk-service/src/tq_service_app/tq_app.py`

### 1.2 Refdata Loader Service

Language: Python

Responsibilities:
- periodic and incremental refdata ingestion from exchange REST APIs
- normalize and upsert into `cfg.instrument_refdata` (PostgreSQL)
- publish `zk.control.refdata.updated` NATS control event on completion

Reference: instrument sync jobs in zkbot/services/zk-refdata-loader

## 2. Data Plane

### 2.1 Strategy Engine Service

Language: Rust (with optional Python worker subprocess via `zk-pyo3-rs`)

Domain core: `zk-engine-rs` + `zk-strategy-sdk-rs`

Responsibilities:
- run one or more strategy instances (Rust-native or Python-embedded)
- consume RTMD, order, balance, position events from NATS
- submit order/cancel commands and queries to OMS via gRPC (`zk.oms.v1.OMSService`)
- emit strategy events, logs, signals, notifications to NATS

Python strategies run as subprocess workers communicating via `zk-pyo3-rs` bindings.

Startup inputs (env vars or CLI args):
- `ZK_STRATEGY_KEY` — strategy definition key
- `ZK_INSTANCE_ID` — unique integer instance id (used for Snowflake order-ID generation; must be unique across all running instances)
- `ZK_NATS_URL`
- `ZK_PILOT_API_URL` — Pilot REST base URL
- `ZK_EXT_SCRIPTS_DIR` — path to strategy script files

Startup sequence (reference: `services/zk-service/src/tq_service_strategy/strat_service.py`):
1. generate `execution_id` locally (UUID)
2. POST `{strategy_key, execution_id}` to Pilot REST `POST /v1/strat/strategy-inst` to register execution and receive strategy config (`symbols`, `account_ids`, `tq_config`, `strategy_script_file_name`, etc.)
3. initialize `TradingClient` (or equivalent) with `instance_id` and resolved account config
4. resolve OMS gRPC endpoint via NATS KV (`svc.oms.<oms_id>`)
5. open gRPC channel to OMS with backoff reconnect
6. subscribe to NATS RTMD and account event topics per configured symbols/accounts
7. start engine event loop

Shutdown:
- publish `StrategyLifecycleNotifyEvent` (STOPPED or EXCEPTIONED) to NATS
- DELETE `/v1/strat/strategy-inst?strategy_key=<key>` to deregister execution from Pilot

### 2.2 OMS Service

Language: Rust

Domain core: `zk-oms-rs` (`OmsCore`, `OrderManager`, `BalanceManager`, `ConfdataManager`)

Responsibilities:
- enforce risk checks and order routing (using `ConfdataManager` / `InstrumentTradingConfig`)
- maintain order/trade/position/balance in-memory state (single writer per instance)
- validate order IDs generated by clients (trading_sdk, UI, or direct API); OMS does not generate IDs — it validates uniqueness only
- resolve and call gateway execution endpoints via `zk.gateway.v1.GatewayService` gRPC
- expose `zk.oms.v1.OMSService` gRPC API
- publish OMS events to NATS (Recorder writes trades and snapshots from these events; OMS does not write to PostgreSQL directly)
- write order/position/balance state to Redis on every state transition (warm-start cache + read replica for Pilot/UI)

Known stubs to complete before production (tracked in `docs/rust-port-plan/07-open-questions-and-non-port-changes.md`):
- `OmsCore::calc_balance_changes_for_report` returns `None` — full spot/margin bookkeeping deferred (Phase 2b)
- `pending_order_reports` cache has no max-size bound
- LRU eviction only removes from `exch_ref_to_order_id`, not `context_cache`

Startup sequence:
Startup sequence by deployment mode:

3-layer mode (Engine + OMS + GW):
1. load `cfg.oms_instance` and `cfg.account_binding` rows from PostgreSQL
2. load per-account instrument/risk config from PostgreSQL via `ConfdataManager`
3. watch NATS KV `svc.gw.*`; resolve gateways for bound accounts by `account_id` and capability
4. establish gRPC clients to each gateway (`zk.gateway.v1.GatewayService`)
5. warm-start from Redis cached open orders/positions for faster readiness
6. run gateway reconciliation pass (`QueryOrder` + `QueryBalance`) to converge to exchange truth
7. register `svc.oms.<oms_id>` in NATS KV with gRPC transport metadata
8. start gRPC server and NATS subscribers

2-layer mode (Engine + OMS with GW plugin):
1. load `cfg.oms_instance`, `cfg.account_binding`, and plugin config from PostgreSQL
2. initialize in-process gateway plugins for bound accounts (implements `GatewayPlugin` trait — direct function calls, no gRPC)
3. warm-start from Redis cached open orders/positions
4. plugin reconciliation pass to live venue state
5. register `svc.oms.<oms_id>` in NATS KV
6. Engine and OMS communicate via **in-process async channel** (e.g. `tokio::sync::mpsc`) instead of gRPC — eliminates loopback network overhead
7. OMS events published to NATS for Recorder/Monitor; strategy callbacks invoked directly via shared channel

All-in-one mode:
1. initialize strategy runtime + OMS core + gateway plugin in one process
2. warm-start from Redis optional cache
3. reconcile from live gateway state
4. all inter-component calls (strategy → OMS, OMS → GW plugin) use **in-process queues or direct function calls** — no gRPC, no NATS on the critical path
5. NATS used only for external event publishing (Recorder, Monitor) and control messages

Periodic tasks (mirrors existing Python intervals):
- order resync every 60s: reconcile orders missing `exch_order_ref` against gateway
- balance resync every 60s or on staleness threshold (> 5 min without update)
- state cleanup every 10 min: evict filled/cancelled orders from in-memory maps

Gateway reconnect handling (mirrors `GW_EVENT_STARTED` logic in Python OMS):
- on `zk.gw.<gw_id>.system` event (gateway restart/reconnect), trigger order and balance resync for all affected accounts

Panic mode:
- Options:
  - Option A (default): block new order placement for specified accounts.
  - Option B: block + cancel all open orders for specified accounts.
  - Option C: block + cancel all open orders + emergency position close workflow.
- `Panic(PanicRequest)` → applies selected panic policy for specified accounts
- `ClearPanic(ClearPanicRequest)` → resumes order placement

### 2.3 Trading Gateway Service

Language: Rust (or Go where venue SDK is Go-only)

One instance per account.

Responsibilities:
- exchange/broker connectivity, signing, and session keepalive
- order/cancel/query execution
- normalize execution reports and publish to NATS:
  - `zk.gw.<gw_id>.report` → `exch_gw.OrderReport`
  - `zk.gw.<gw_id>.balance` → `exch_gw.BalanceUpdate`
  - `zk.gw.<gw_id>.system` → `exch_gw.GwSystemEvent`
- self-register in NATS KV `svc.gw.<gw_id>` with heartbeat (5s interval, 20s TTL)

Gateway capability flags (migrated from MongoDB `refdata_gw_config` to `cfg.gateway_instance`):
- `supports_batch_order`, `supports_batch_cancel`
- `supports_order_query`, `supports_position_query`
- `supports_trade_history_query`, `supports_fee_query`
- `cancel_required_fields` — list of fields required in cancel requests

Startup:
1. fetch account secrets from Vault (`kv/trading/gw/<account_id>/api_key` etc.)
2. initialize venue session and validate connectivity
3. register `svc.gw.<gw_id>` in NATS KV with `account_ids`, capabilities, and gRPC transport
4. start heartbeat loop (5s)

Trading Simulator is a specialized gateway type implementing the same `GatewayService` interface.

### 2.4 Market Data Gateway Service

Language: Rust or Python (one instance per venue)

Responsibilities:
- venue market data ingestion and normalization
- dynamic symbol/channel subscription: reads active subscription set from PostgreSQL `cfg.mdgw_subscription` table (written by Pilot admin UI) and watches for changes via `zk.control.refdata.updated` or a dedicated `zk.control.mdgw.<venue>.reload` event
- publish normalized streams:
  - `zk.rtmd.tick.<venue>.<instrument_exch>` → `rtmd.TickData`
  - `zk.rtmd.kline.<venue>.<instrument_exch>.<interval>` → `rtmd.Kline`
  - `zk.rtmd.funding.<venue>.<instrument_exch>` → `rtmd.FundingRate`
  - `zk.rtmd.orderbook.<venue>.<instrument_exch>` → `rtmd.OrderBook`

Topic names are deterministic — strategies and `trading_sdk` construct them locally.
No ODS lookup is ever needed for RTMD channel resolution.

Reference: `services/zk-service/src/tq_service_app/tq_rtmd_registry_mgr.py`

### 2.5 Trading Recorder + Reconciliation Service

Language: Python (or Rust for high-volume ingest paths)

Can run as multiple instances (partitioned by `oms_id` or `strategy_id`).

Responsibilities:
- subscribe to OMS and strategy NATS event topics
- write immutable event records to MongoDB (`zk_events` database)
- write raw OMS trades to PostgreSQL `trd.trade_oms` when `OrderUpdateEvent.last_trade` is present
- write hourly balance and position snapshots to `trd.balance_snapshot` and `trd.position_snapshot` from `oms.PositionUpdateEvent` stream
- run scheduled reconciliation jobs that:
  - fetch OMS trade records (`trd.trade_oms`) and compare against exchange trade history from gateway
  - produce reconciled results into `trd.trade_recon` and log the run in `mon.recon_run`
  - flag `OMS_ONLY`, `EXCH_ONLY`, and `AMENDED` discrepancies

Reference implementations:
- `tq_service_app/tq_strat_recorder.py`
- `tq_service_oms/oms_recorder.py`
- `tq_service_oms/order_event_recorder.py`
- `tq_service_oms/oms_balance_collector.py`
- `tq_service_oms/post_trade_handler.py`

### 2.6 Trading Monitor Service

Language: Python

Global service (single logical instance, optionally replicated).

Responsibilities:
- subscribe to OMS `order_update` and `balance_update` NATS streams
- evaluate configurable risk/operational rules:
  - rejection rate watcher (`order_updates_monitor.py` `reject_watcher`)
  - fill rate and volume limit watcher
  - channel staleness watcher (`nats_channel_monitor.py`)
  - open order aging watcher (`open_order_monitor.py`)
  - PnL limit watcher (`pnl_monitor.py` + `pnl_calculator.py`)
- publish structured notifications to `zk.notify.*`
- send external alerts (Slack, PagerDuty etc.) via notification hub

Reference implementations:
- `tq_service_riskmonitor/order_updates_monitor.py`
- `tq_service_riskmonitor/balance_updates_monitor.py`
- `tq_service_riskmonitor/pnl_monitor.py` + `pnl_calculator.py`
- `tq_service_riskmonitor/nats_channel_monitor.py`
- `tq_service_riskmonitor/open_order_monitor.py`
- `tq_service_helper/notification_hub.py`
