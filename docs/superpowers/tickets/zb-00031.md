# zb-00031 Unify venue manifest-root configuration across refdata, gateway, RTMD, and operator tooling

- Ticket: zb-00031
- Date: 2026-04-19
- Time: 13:06 +08
- Domain: design, gw, mdgw, refdata, pilot
- Reporter: agent
- Status: open

## Description

The post-refactor repo now has an explicit package-based code-loading contract, but the on-disk venue manifest/schema root is still configured through inconsistent env var names and host-specific expectations. Refdata's venue registry expects ZK_VENUE_INTEGRATIONS_DIR, while GW/RTMD configs and Make targets still use ZK_VENUE_MANIFEST_ROOT. This splits one repository boundary into multiple operator contracts and weakens the current code structure.

## Symptoms

- zk_refdata_svc.venue_registry requires ZK_VENUE_INTEGRATIONS_DIR or an explicit set_integrations_dir(...) override
- gw-svc config still reads ZK_VENUE_MANIFEST_ROOT into venue_root and carries that field into runtime config
- rtmd-gw-svc config also reads ZK_VENUE_MANIFEST_ROOT
- Makefile gateway and RTMD targets export ZK_VENUE_MANIFEST_ROOT, not the refdata contract name
- dependency-contract.md documents ZK_VENUE_INTEGRATIONS_DIR as the manifest-root env var contract, so active runtime code and docs already disagree

## Impact

Operators and agents now have to remember two different env var names for the same on-disk venue manifest boundary depending on which host they are launching. That increases setup errors, weakens bootstrap consistency, and makes the intended code structure less obvious: one manifest root should look like one contract.

## Proposed Options Or Fixes

1. Pick one canonical manifest-root setting for all hosts, ideally matching the docs, and migrate refdata, gw, rtmd, Make targets, and any pilot wiring to that single name
2. If a rename is too disruptive, add a short transitional alias period with one canonical name in docs and one compatibility alias in code, then remove the alias in a follow-up cleanup stage
3. Document clearly that manifest/schema discovery and Python package import are separate concerns: one explicit env var for on-disk manifests, installed-package imports for Python modules

## Open Questions

- Should the canonical name be ZK_VENUE_INTEGRATIONS_DIR, ZK_VENUE_MANIFEST_ROOT, or a third name that better matches the repo boundary?
- Should pilot-service and other Java-side launch surfaces pass this setting through explicitly for spawned hosts, or should each runtime host resolve it independently from its own environment?
