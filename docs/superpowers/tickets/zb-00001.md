# zb-00001 PUT /v1/topology/services/{kind}/{id} needs schema validation of providedConfig

- Ticket: zb-00001
- Date: 2026-03-24
- Time: 17:36 +08
- Domain: pilot
- Reporter: agent
- Status: resolved

## Description

The new PUT endpoint for updating service config persists providedConfig directly without running it through the JSON schema validator. Should validate against the resolved active schema (service-kind schema + venue-capability schema for GW/MDGW) before allowing the update.

## Symptoms

- The update service config endpoint persists providedConfig without validating it against the resolved active JSON schema. Invalid config can be saved to the DB.

## Impact

Operators could save malformed config that fails at runtime bootstrap, causing service startup failures that are hard to diagnose.

## Proposed Options Or Fixes

1. 1. Add schema validation in TopologyService.updateService() using resolveSchemaProvenance + JsonSchemaFactory. 2. For GW/MDGW, validate against both service-kind and venue-capability schemas. 3. Return 422 with validation errors if config doesnt match schema.

## Open Questions

- 1. Should partial config updates (PATCH semantics) be supported, or only full config replacement? 2. Should the endpoint also validate venue_config sub-object separately against the venue-capability schema?
