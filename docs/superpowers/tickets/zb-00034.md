# zb-00034 Pilot REFDATA bootstrap: stop creating per-venue logical_instance rows; aggregate venues into `{venues: [...]}` payload

- Ticket: zb-00034
- Date: 2026-04-23
- Time: 22:20 +08
- Domain: pilot, refdata, bootstrap
- Reporter: agent
- Status: resolved

## Description

The REFDATA service model is: **one logical_instance per env, serving N venues.** Each venue has its own row in `cfg.refdata_venue_instance` (operator-authored `provided_config`); the refdata service instance itself lives in `cfg.logical_instance` once per env (`refdata_<env>_1`).

Two coordinated defects break this model today:

1. **`TopologyService.createRefdataVenueInstance` wrongly creates a `logical_instance` row per venue.** See [TopologyService.java:489-491](zkbot/java/pilot-service/src/main/java/com/zkbot/pilot/topology/TopologyService.java#L489-L491). Each `POST /v1/topology/refdata-venues` inserts `(request.logicalId(), "REFDATA", ...)` into `logical_instance`, polluting the topology namespace with venue-scoped rows that are not services.

2. **Pilot's bootstrap payload for REFDATA is single-row.** [DesiredConfigRepository.java:61-62](zkbot/java/pilot-service/src/main/java/com/zkbot/pilot/config/DesiredConfigRepository.java#L61-L62) looks up `cfg.refdata_venue_instance WHERE logical_id = ?` and returns that one venue's `provided_config` verbatim. But `zk-refdata-svc` expects a multi-venue aggregate: `{"venues": [{"venue": ..., "enabled": ..., "config": {...}}, ...]}` (see [config_loader.py:54-69](zkbot/python/services/zk-refdata-svc/src/zk_refdata_svc/config_loader.py#L54-L69)).

Net effect: there is no working pilot-mode path from the operator-authored venue rows to a loading refdata-svc.

## Symptoms

- Current state in dev DB (2026-04-23) shows the pollution directly:
  ```
  cfg.logical_instance (instance_type=REFDATA):
    refdata_dev_1       (legitimate service instance)
    refdata_ibkr_dev_1  (rogue — created by POST /v1/topology/refdata-venues)

  cfg.refdata_venue_instance:
    refdata_ibkr_dev_1  / ibkr
    refdata_oanda_dev_1 / oanda
    refdata_okx_dev_1   / okx
    refdata_sim_dev_1   / simulator
  ```
  Older venue rows (oanda/okx/sim) predate the defect — they have no matching `logical_instance` row. Any new `POST /v1/topology/refdata-venues` call now inserts a rogue `logical_instance` row.

- `make refdata-run-pilot ZK_BOOTSTRAP_TOKEN=...` with `ZK_REFDATA_LOGICAL_ID=refdata_dev_1` fails:
  ```
  bootstrap rejected: status=ERROR error=No desired config found for refdata_dev_1
  (instance_type=REFDATA). Ensure the service-specific table has a config row.
  ```
  Reason: DesiredConfigRepository queries `cfg.refdata_venue_instance WHERE logical_id = ?` — that table is per-venue, so `refdata_dev_1` has no row. `refdata_dev_1` is the *correct* service identity; the lookup shape is wrong, not the logical_id.

- Switching `ZK_REFDATA_LOGICAL_ID` to a per-venue id (e.g. `refdata_ibkr_dev_1`) would pass the lookup only because of defect (1). Even then, Pilot returns the raw IBKR-only dict; `zk-refdata-svc` reads `venues_data = runtime_config.get("venues", [])` → 0 enabled venues, refresh jobs disabled.

## Impact

- Pilot-mode refdata is effectively non-functional. Operators cannot use the documented Pilot onboarding flow to bring up refdata for a new venue.
- Topology is polluted with non-service `logical_instance` rows. Any code that lists REFDATA services (topology views, session listings, service-kind filters) sees the venue rows and treats them as services.
- Today's only working path is direct mode with `ZK_REFDATA_VENUES=...` + a local JSON file at `devops/refdata_venue_configs.json`. That bypasses Pilot entirely and duplicates config.
- Blocks the clean end-to-end onboarding story for IBKR (and any future venue) through the Pilot UI.

## Proposed Options Or Fixes

The fix has two coordinated parts.

### Part A — stop creating per-venue logical_instance rows

- Remove the `repository.createLogicalInstance(request.logicalId(), "REFDATA", ...)` call at [TopologyService.java:491](zkbot/java/pilot-service/src/main/java/com/zkbot/pilot/topology/TopologyService.java#L491). Venue rows live *only* in `cfg.refdata_venue_instance`.
- Ensure exactly one `cfg.logical_instance` row with `instance_type='REFDATA'` per env (e.g. `refdata_<env>_1`). Seed via a new Flyway migration `V*__seed_refdata_logical_instance.sql`.
- Data migration to clean up existing rogue rows: delete `logical_instance` rows where `instance_type='REFDATA'` AND `logical_id` exists in `cfg.refdata_venue_instance`. In the current dev DB: delete `refdata_ibkr_dev_1` from `logical_instance` (keeping the row in `refdata_venue_instance`).
- FK / constraint sweep: nothing should reference `logical_instance.logical_id` with a per-venue id.

### Part B — aggregate bootstrap payload for REFDATA

- Special-case `instance_type=REFDATA` in `DesiredConfigRepository.getDesiredConfig`: ignore the `logical_id` arg and instead read all enabled rows from `cfg.refdata_venue_instance WHERE env = ?`, returning `{"venues": [{"venue": <venue>, "enabled": <enabled>, "config": <provided_config>}, ...]}`. Env comes from the bootstrap request (already on `BootstrapRegisterRequest`).
- `config_version`/`config_hash` become aggregates — `max(config_version)` + a synthetic sha256 over the serialized `{venues: [...]}` payload.
- `setDesiredConfig` for REFDATA is already deprecated on that path; per-venue writes flow through `TopologyService.updateRefdataVenueConfig`. No change needed.

### Rejected alternatives

- **One refdata-svc per venue.** Would require changing `zk-refdata-svc` to drop the `venues` wrapper and running N processes. More ops surface; loses the existing multi-venue design.
- **Sidecar resolver in refdata-svc.** Keep single-row bootstrap but have refdata-svc pull sibling rows from Postgres after bootstrap. Two config paths → split authority.

## Open Questions

- When a venue row changes, should Pilot push a `zk.control.REFDATA.<logical_id>.reload` notification, or do we accept bootstrap-on-restart as the only propagation path for now? Today `zk-refdata-svc` does not subscribe to any reload subject — see [publish_changes.py:11](zkbot/python/services/zk-refdata-svc/src/zk_refdata_svc/jobs/publish_changes.py#L11), only publishes `zk.control.refdata.updated`. Out of scope here; worth filing a follow-up.
- `secret_refs` extraction ([BootstrapService.java:211-212](zkbot/java/pilot-service/src/main/java/com/zkbot/pilot/bootstrap/BootstrapService.java#L211-L212)) currently resolves via `ConfigSchemaLocator` keyed on `(logical_id, instance_type)`. For an aggregate payload the locator must resolve per-venue schemas and fold their secret_refs together with venue-scoped JSON pointer prefixes (e.g. `/venues/0/config/secret_ref`). Design note needed.
- Seeding the one-per-env `refdata_<env>_1` `logical_instance` row: Flyway migration, Pilot startup idempotent insert, or explicit operator-owned `POST /v1/topology/services/REFDATA`? First is cleanest for dev/prod parity.
- Should `cfg.refdata_venue_instance.logical_id` be renamed (e.g. to `venue_instance_id`) to remove the confusing naming collision with `logical_instance.logical_id`? Nice-to-have; not blocking.
