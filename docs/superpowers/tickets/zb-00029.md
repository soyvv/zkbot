# zb-00029 Define a consistent Rust-Python dependency and packaging contract for local dev and CI/CD

- Ticket: zb-00029
- Date: 2026-04-18
- Time: 14:55 +08
- Domain: design, engine, strategy, gw, mdgw, pilot
- Reporter: agent
- Status: open

## Description

The repo now has multiple real Rust↔Python dependency edges, but their resolution model is inconsistent across local development, embedded runtime execution, and future CI/CD or deployment packaging. Some flows rely on sys.path injection, explicit PYTHONPATH, repo-relative search paths, or workspace-relative binary discovery. A follow-up to zb-00028 should define one consistent contract for how Rust consumes Python code, how Python consumes Rust-built artifacts, and how all environments local dev, CI/CD, and production use the same explicit packaged or artifact-backed resolution model with no dev-only fallback path semantics.

## Symptoms

- zk-pyo3-bridge initializes embedded Python by mutating sys.path and probing venue-local .venv site-packages, with optional ZK_VENUE_VENV override
- zk-strategy-host-rs injects python_search_path into sys.path for python-wrapper strategies before importing the module
- engine and strategy launch paths currently depend on explicit PYTHONPATH and interpreter wiring in Makefile targets instead of a packaged dependency contract
- Pilot backend still resolves zk-engine-svc using multiple repo-relative fallback paths that mainly fit local dev layouts
- zkstrategy_research and several tools/docs still describe imports via PYTHONPATH or sys.path extension to reach zkbot code
- refdata and tooling paths also use sys.path insertion, which makes package boundaries and deployment-time dependencies ambiguous

## Impact

Without an explicit dependency model, the current setup works mainly in the dev workspace but remains fragile for CI, packaging, and deployment. Embedded Python hosts, Python strategy execution, venue adaptors, Pilot launches, and research workflows can each succeed for different reasons, making failures hard to predict and making it unclear which path-based behaviors are intentional contracts versus local convenience hacks.

## Proposed Options Or Fixes

1. Classify every Rust-Python edge as one of the explicit supported contract modes that remain after this refactor, and document which mode each subsystem uses, but do not keep a local source-checkout fallback mode as a supported runtime contract
2. Require packaged or artifact-backed resolution in local dev, CI/CD, and production for embedded boundaries: Python modules should be installed from packages or explicit package roots managed by uv, and Rust artifacts should be referenced through explicit build outputs rather than cwd-relative discovery
3. Remove dependence on raw PYTHONPATH injection, ad hoc sys.path mutation, venue-local venv probing, and repo-relative binary guessing across all supported environments; do not keep dev-only escape hatches once the packaging contract exists
4. Define a single explicit resolver contract for embedded Python hosts such as zk-pyo3-bridge and python-wrapper, based on configured interpreter plus installed importable modules only
5. Introduce a dependency matrix and validation checks that cover local dev, CI, and deployment for Rust consuming Python packages, Python consuming Rust extensions or binaries, and Java or Pilot invoking Rust services

## Open Questions

- Should venue adaptors be packaged as installable Python distributions per venue, or should the supported contract remain an explicit package-root mount with manifest resolution backed by uv-managed installs?
- How should Rust-built Python extensions such as zk-pyo3-rs be versioned and published for downstream Python consumers once local maturin develop usage is removed?
- Should Pilot resolve engine binaries only from configured artifact paths once the packaging contract is in place, with all local repo-relative candidates removed immediately?
- Should local development use only the same packaged-artifact contract as CI and production even when that temporarily blocks work until packaging metadata is fixed, or is there any remaining exception that still needs to be named explicitly?
