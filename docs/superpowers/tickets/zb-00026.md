# zb-00026 RTMD gateway event loop exits on recoverable adapter errors such as OANDA next_event TimeoutError

- Ticket: zb-00026
- Date: 2026-04-14
- Time: 23:14 +08
- Domain: mdgw, gw
- Reporter: codex
- Status: resolved

## Description

The RTMD gateway event loop in zk-rtmd-gw-svc exits on any adapter error other than ChannelClosed. OANDA RTMD produced recoverable TimeoutError failures on April 12, and the event loop stopped publishing until the process was manually restarted. This makes the RTMD service fragile to transient idle or venue-side timeouts.

## Symptoms

- rtmd-oanda-demo-pilot log on April 12 shows event loop error with venue error: next_event: TimeoutError and then no recovery inside the same process
- zk-rtmd-gw-svc main event loop breaks on any Err(e) path in main.rs, not just terminal channel-close conditions
- After manual restart, OANDA RTMD resumed working, indicating the failure was process-liveness fragility rather than persistent bad config

## Impact

A transient adapter timeout can silently stop RTMD event publication for a venue while the service process and KV registration remain alive. Operators may believe MDGW is healthy even though market data flow has died. Strategies and downstream consumers can run on stale or missing data until someone restarts the service.

## Proposed Options Or Fixes

1. Handle known recoverable adapter errors such as TimeoutError by logging, applying bounded backoff, and continuing the event loop instead of breaking immediately
2. Add adaptor-side hardening so OANDA next_event preserves blocking semantics across idle weekends and does not surface TimeoutError as a fatal error to the Rust host
3. Introduce a health counter / last-event watchdog in zk-rtmd-gw-svc so the service can emit degraded-state logs or system events when the adapter repeatedly errors without data
4. Reserve event-loop exit for terminal conditions such as ChannelClosed, explicit shutdown, or repeated unrecoverable errors after a bounded retry budget

## Open Questions

- Should recoverable-error handling live only in zk-rtmd-gw-svc, or should Python RTMD adaptors be required to mask idle timeouts and maintain the blocking next_event contract
- Do we want a generic retry classifier in zk-rtmd-rs / bridge code for Python venues, or a narrow OANDA-specific fix first
- Should MDGW service registration expose degraded health when the event loop is alive but repeatedly failing

## Resolution

- Resolved At: 2026-04-14 23:17 +08
- Summary: Fixed RTMD gateway event loop in zk-rtmd-gw-svc/src/main.rs to retry on transient adapter errors with exponential backoff (1s to 30s) instead of breaking immediately. Only exits after 10 consecutive errors or ChannelClosed. Resets error count on successful event. The bridge-side fix (rtmd_adapter.rs retry loop for TimeoutError) was already in place from zb-00020; this hardens the service-level loop against any remaining transient errors. Both build targets (default + python-venue) pass.
