# zb-00008 Improve OKX reject reason logging and normalization

- Ticket: zb-00008
- Date: 2026-03-26
- Time: 00:15 +08
- Domain: gateway, okx, observability, orders
- Reporter: agent
- Status: resolved

## Description

OKX order rejects are not currently operator-readable enough.

The gateway logs show the top-level adapter failure, but the most useful venue
reason is not surfaced consistently in structured logs or preserved clearly in the
synthetic rejection report path sent downstream.

Need to make an OKX reject explain itself without requiring code inspection.

## Symptoms

- gateway log shows generic text such as `All operations failed`
- operators cannot quickly tell whether the reject was caused by margin,
  account mode, instrument constraints, or invalid order parameters
- the rejection context visible in OMS is weaker than the raw venue failure

## Impact

- slower incident debugging
- harder onboarding for new OKX demo/live accounts
- unnecessary dependence on reconcile or manual diagnosis for straightforward
  venue rejects

## Proposed Options Or Fixes

1. Parse and preserve structured OKX error payload fields at the REST client layer.
2. Log per-order reject context with `order_id`, `clOrdId`, endpoint, HTTP status,
   OKX `code`, OKX `msg`, and any item-level reject details.
3. Normalize the best available venue reject reason into
   `rejection_info.error_message` for downstream OMS handling.
4. Ensure synthetic gateway reject reports preserve both normalized and raw venue
   context where possible.
5. Add targeted tests for reject extraction and normalization.

## Acceptance Criteria

- a rejected OKX order has actionable reason text in gateway logs
- the synthetic reject report sent to OMS carries useful venue reason text
- logs and downstream rejection payloads can be correlated by `order_id`
