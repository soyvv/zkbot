# zb-00035 refresh_sessions spawns a fresh loader and races to reconnect

- Ticket: zb-00035
- Date: 2026-04-27
- Time: 23:30 +08
- Domain: refdata
- Reporter: agent
- Status: resolved

## Description

The periodic refresh_sessions job calls instantiate_refdata_loader() which creates a NEW IbkrRefdataLoader instance and a NEW IBKR connection on every cycle. This races with the main refresh_refdata loop holding its own connection on the same client_id and intermittently fails.

## Symptoms

- Repeating ERROR in zk-refdata-svc log: '[ibkr] failed to load sessions from adaptor' followed by 'ConnectionError: failed to connect to TWS/IB Gateway:'.
- Observed at 5-minute periodic intervals after the main refresh loop completes (15:43:56, 15:49:04, 15:54:12, ...).
- Root cause: refresh_sessions builds a fresh loader each tick instead of reusing the one held by refresh_refdata; the two compete for the same IBKR client_id.

## Impact

Market session state for IBKR (open/closed) is never updated by refresh_sessions, breaking strategies that gate trading on session_state. Strategies that read cfg.market_session_state see stale or fallback 'closed' values for IBKR.

## Proposed Options Or Fixes

1. Share the loader instance between refresh_refdata and refresh_sessions via the new RefreshCoordinator (it already owns per-venue locks; add a per-venue cached loader).
2. Or: have refresh_sessions read session_state from the existing loader's contract_details_cache via a method on the coordinator instead of spawning a new loader.

## Open Questions

- Should the coordinator own a long-lived loader per venue, or spin up + tear down per refresh? IBKR's reconnect-with-same-client-id constraint pushes toward long-lived.

## Resolution

- Resolved At: 2026-04-29 00:24 +08
- Summary: Routed refresh_sessions through RefreshCoordinator. Coordinator now caches the loader instance per venue (lazy init via _get_loader). Both refresh_all_periodic and the new load_market_sessions_for(venue) reuse the cached loader and serialize under the same per-venue asyncio.Lock. Eliminates the race where every periodic tick spawned a fresh IbkrRefdataLoader competing for the same client_id with the still-active main loader. refresh_sessions.py signature changed to (coordinator, repo, venues); main.py wiring updated. 113 refdata-svc tests green; coordinator.test_periodic_serializes_with_manual still passes — the lock semantics are unchanged. Side benefit: contract_details_cache persists across refresh + sessions calls in the same coordinator instance, so sessions can avoid re-qualifying contracts when the cache is warm.
