# zb-00017 Pilot account live OMS queries have no deadline and can leave Trading UI requests pending indefinitely

- Ticket: zb-00017
- Date: 2026-03-28
- Time: 22:29 +08
- Domain: pilot
- Reporter: Codex
- Status: resolved

## Description

The Trading UI Open Orders tab can get stuck because Pilot forwards /v1/accounts/{id}/orders/open to OMS through a blocking gRPC stub with no deadline. When OMS or a downstream gateway stalls, the HTTP request remains pending and the operator-facing panel appears frozen.

## Symptoms

- Browser network panel shows /v1/accounts/8002/orders/open stuck in pending state for a long time
- Trading Open Orders tab appears blocked waiting on one unresolved request
- Pilot AccountService.getOpenOrders() performs a synchronous OMS query with no timeout handling
- OmsGrpcClient queryOpenOrders/queryPosition/queryBalances all use blocking stubs without deadlines

## Impact

A single stalled OMS query can hang Pilot HTTP responses indefinitely, making trading panels feel frozen and tying up backend request handling threads. The same pattern also risks hanging positions and balances endpoints, not just open orders.

## Proposed Options Or Fixes

1. Add explicit per-call deadlines to OMS gRPC query methods in OmsGrpcClient
2. Catch deadline-exceeded and unavailable errors in AccountService and return a controlled timeout/error response instead of hanging
3. Optionally add separate timeout budgets for open orders, positions, and balances if they have different latency expectations
4. Surface timeout/error state in the UI live tabs so operators see a bounded failure rather than a permanent spinner

## Open Questions

- Should Pilot return partial empty data on OMS timeout, or a clear timeout error for live account queries?
- Should OMS query deadlines be centralized in config so live panel latency can be tuned per environment?

## Resolution

- Resolved At: 2026-03-28 22:33 +08
- Summary: Added 5-second gRPC deadline to all OMS stubs via withDeadlineAfter() in OmsGrpcClient.getStub(). Wrapped getBalances(), getPositions(), and getOpenOrders() in AccountService with StatusRuntimeException catch blocks that log a warning and return empty lists on timeout/unavailable. HTTP requests are now bounded at ~5s worst case instead of hanging indefinitely.
