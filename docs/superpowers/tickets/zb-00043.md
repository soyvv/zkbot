# zb-00043 Rust OMS uses '== SPOT' as spot-like check; should match Python SPOT_LIKE_TYPES = {SPOT, ETF, STOCK}

- Reporter: agent

## Description

Canonical instrument-type classification in zk_utils.instrument_utils.py defines SPOT_LIKE_TYPES = {SPOT, ETF, STOCK} (full-value trading, position tracked by base_asset) and MARGIN_TYPES = {PERP, FUTURE, CFD, OPTION}. The Rust OMS code consistently uses 'instrument_type == SPOT_INSTRUMENT_TYPE' (proto value 1) as the spot-like predicate, missing ETF and STOCK.

## Symptoms

- 5+ call sites hardcode SPOT-only check: oms_core_v2.rs:1307,1366,647; oms_core.rs:829,1135; utils.rs:59,84; nats_handler.rs:178.
- For an IBKR equity position with instrument_type=STOCK, OMS treats it as 'non-spot' and routes through the position branch (which works for the user's case but mis-models the underlying semantics).
- If a future code path uses '== SPOT' to gate spot-specific bookkeeping (e.g. PnL accrual, margin check), STOCK/ETF will silently be treated as derivatives.

## Impact

Subtle: today's IBKR position flow happens to work because the position branch is correct for equities. But the model is wrong (equities aren't derivatives), and a future change to the position branch could regress equities silently.

## Proposed Options Or Fixes

1. Introduce 'pub const SPOT_LIKE_TYPES: &[i32] = &[InstTypeSpot, InstTypeEtf, InstTypeStock]' in zk-oms-rs/src/utils.rs and a helper 'pub fn is_spot_like(t: i32) -> bool'. Replace every '== SPOT_INSTRUMENT_TYPE' with 'is_spot_like(t)'. Then choose: do equities route to balance store (per Python model) or position store (per current behavior)?
2. Or: split balance vs position routing on a different signal (e.g. base_asset != quote_asset for derivatives, base_asset present + quote_asset for spot-like). Larger refactor.
3. Defer: leave the current asymmetry, document it, and revisit when an equity-specific bookkeeping bug is reported.

## Open Questions

- Where should equities live in the OMS model — balance store (full-value, asset-tracked) or position store (instrument-tracked)? Today's code routes them to positions due to 'not equal SPOT'. The Python canonical model says balances. Decision affects strategy SDK + UI semantics.

## Resolution

- Resolved At: 2026-04-29 00:26 +08
- Summary: Added Rust mirror of zk_utils.instrument_utils SPOT_LIKE_TYPES / MARGIN_TYPES + is_spot_like(t)/is_margin(t) helpers in zk-oms-rs/src/utils.rs. 5 new unit tests cover the type-classification logic. Bare SPOT_INSTRUMENT_TYPE constant kept (not removed) for callers that explicitly want SPOT-only semantics; doc comment now points new code at is_spot_like. Migration of the 5+ existing '== SPOT_INSTRUMENT_TYPE' call sites (oms_core_v2.rs:1307,1366,647 etc.) is intentionally deferred — it requires a product/design decision about whether equities (STOCK/ETF) belong in the balance store (per Python canonical model) or the position store (per current behavior, matching IBKR's UI 'Holdings' framing). Today's IBKR position flow happens to work because STOCK!=SPOT routes through the position branch; that asymmetry is documented in the call-site code comments. Once the design call is made, switching '== SPOT_INSTRUMENT_TYPE' to 'is_spot_like(t)' is a one-line change per call site. 77 zk-oms-rs tests green.
