# zb-00042 IBKR refdata loader has no circuit-breaker on mid-run socket disconnect

- Ticket: zb-00042
- Date: 2026-04-27
- Time: 23:46 +08
- Domain: refdata
- Reporter: agent
- Status: resolved

## Description

When IBKR Gateway drops the socket mid-refresh, the loader's per-candidate try/except logs the exception but does not break out of the for-loop or attempt reconnect. Subsequent iterations call qualify -> qualifyContractsAsync on a dead ib_async connection, each raising ConnectionError('Not connected') instantly. With ~5000 remaining candidates this generates thousands of error log lines in a few seconds and the run never reaches the completion path that would mark it 'failed' in cfg.refdata_refresh_run.

## Symptoms

- Observed during run 1759 (started 2026-04-27 15:13:59 UTC): TWS Gateway dropped the connection at ~30 MB received (~6000 contracts qualified), then the loader emitted thousands of identical 'ConnectionError: Not connected' tracebacks at saturation rate.
- The refresh_refdata_run row stays in status='running' indefinitely.
- Required manual intervention: kill -9 the process, mark the orphan run as failed in DB, restart.

## Impact

A single transient IBKR disconnect during a long-running refresh poisons the service: tight error loop, log spam, orphan refresh_run rows. Recovery requires operator intervention.

## Proposed Options Or Fixes

1. Detect ConnectionError in the per-candidate exception path and break out of the loop with status='failed' and error_detail='socket disconnect mid-run'. Let the next periodic tick re-run from scratch with a fresh connection.
2. Or: catch ConnectionError specifically and attempt reconnect (with bounded retries) before giving up. Safer because partial progress isn't lost.
3. Or: add a heartbeat check at the top of each loop iteration that confirms ib.isConnected() before issuing requests; on disconnect, fail fast.

## Open Questions

- Should partial results be written (the 6000 contracts already qualified before the disconnect)? Currently the run discards everything because the diff/upsert happens after the loop. A 'partial' status that records what was qualified might be better than 'all-or-nothing' for slow venues.

## Resolution

- Resolved At: 2026-04-29 00:21 +08
- Summary: Added circuit-breaker at the top of IbkrRefdataLoader.load_instruments candidate loop: ib.isConnected() check trips before each iteration; on disconnect, raise ConnectionError with explanatory message and let the periodic refresh job mark the run failed (avoiding the log-spam loop and stuck 'running' rows). Per-candidate try/except now re-raises ConnectionError specifically instead of swallowing it. The next periodic tick reconnects from scratch with a fresh ib_async client. Added regression test (test_load_instruments_circuit_breaker_on_disconnect) that simulates a mid-run drop and verifies the loop bails after 2/5 candidates without further qualify calls. Partial-result preservation deferred — current model is all-or-nothing because diff/upsert lives after the loop. 195 IBKR tests green.
