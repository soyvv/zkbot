# zb-00004 Implement live gateway position query and OMS position reconcile

- Ticket: zb-00004

- Time: 23:55 +08
- Domain: oms, gateway, reconcile, runtime
- Reporter: agent
- Status: open

## Description

The position-reconcile half of the OMS position/balance refactor is not operational.

The gateway service exposes `QueryPosition`, and venue adapters such as OKX already
implement venue position queries, but `zk-gw-svc` still returns an empty
`PositionResponse`. OMS startup reconcile calls only `QueryAccountBalance`, and periodic
`PositionRecheck` is still a no-op.

Need to complete the live exchange-position refresh path so OMS can anchor managed
position quantity against exchange truth.

## Symptoms

- `GatewayService.QueryPosition` exists but returns an empty response
- OMS startup reconcile refreshes balances but not exchange positions
- periodic `PositionRecheck` runs but does nothing
- reconcile state exists in data structures but is not driven by live refreshes

## Impact

- OMS-managed position quantity can drift without repair
- canonical position semantics in the refactor doc are not trustworthy in runtime terms
- startup after restart cannot rebuild exchange position anchors

## Proposed Options Or Fixes

1. Implement `zk-gw-svc` `QueryPosition` using `adapter.query_positions()`.
2. Extend OMS startup reconcile to query both balance and position state for connected gateways.
3. Implement `OmsCoreV2::process_position_recheck()` to compare managed vs exchange positions and update reconcile status.
4. Persist refreshed exchange position snapshots during startup reconcile and periodic recheck flows.
5. Add regression coverage for startup reconcile and periodic position recheck behavior.

## Acceptance Criteria

- `QueryPosition` returns venue-backed positions from live gateways.
- OMS startup reconcile refreshes exchange positions as well as balances.
- `PositionRecheck` is no longer a no-op.
- Reconcile metadata changes when managed and exchange quantities diverge.
