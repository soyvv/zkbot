# zb-00022 Registered services rely on narrow shutdown paths, delaying KV cleanup and Pilot liveness updates

- Ticket: zb-00022
- Date: 2026-04-13
- Time: 21:53 +08
- Domain: design, pilot, gw
- Reporter: codex
- Status: resolved

## Description

Registered services such as OMS and GW already call ServiceRegistration::deregister() on their clean shutdown path, but the shutdown handling is duplicated per service and is currently centered on tokio::signal::ctrl_c(). When a process exits through other common paths such as SIGTERM, SIGHUP, shell/session teardown, or other non-ctrl_c termination flows, proactive KV deletion may not happen and discovery falls back to heartbeat TTL expiry. This makes Pilot and other discovery consumers slow to notice that a service has stopped.

## Symptoms

- After a service is stopped from the terminal or session context, Pilot can take noticeably longer than expected to reflect that the service is gone
- KV cleanup and shutdown-selection logic are copy-pasted across OMS, GW, and RTMD GW main functions instead of being enforced by shared infra
- Current registration semantics proactively deregister only on the explicit clean path the service implements; otherwise stale registration visibility lasts until lease TTL expires

## Impact

Operational liveness in Pilot becomes laggy and inconsistent with real process state. Late removal of dead service registrations can mislead operators, delay failover or restart workflows, and create avoidable confusion during local dev and production operations.

## Proposed Options Or Fixes

1. Add a shared shutdown helper in zk-infra-rs for registered services that waits on common termination signals such as SIGINT, SIGTERM, and optionally SIGHUP, and proactively calls deregister() on clean signal-driven shutdown
2. Refactor OMS, GW, and RTMD GW to use the shared infra shutdown helper instead of duplicating tokio::select! shutdown logic in each service main
3. Preserve current fencing semantics: on wait_fenced() the service should terminate without deleting the KV key, because ownership may already have moved to another instance
4. Document that TTL expiry remains the crash-path fallback for kill -9 and abnormal termination, but normal signal-driven shutdown should remove the KV entry immediately

## Open Questions

- Should the shared helper live as a method on ServiceRegistration or as a separate infra utility for service mains?
- Do we want to treat SIGHUP as a clean deregister-and-exit path for terminal/session teardown on Unix?
- Should the infra helper also standardize shutdown logging and return a structured shutdown reason such as signal, fenced, or server exit?

- Notes:
- Current services already call deregister() on their clean ctrl_c path, so the issue is not missing cleanup entirely; it is that signal handling is narrow and duplicated
- Ctrl+D is EOF on stdin, not SIGINT, so terminal/session-driven exits may bypass the current ctrl_c-only handling
- A shared infra-level shutdown contract is a better ownership boundary than repeating signal-selection logic in every service

## Resolution

- Resolved At: 2026-04-13 22:39 +08
- Summary: Added ServiceRegistration::wait_shutdown() to zk-infra-rs that handles SIGINT, SIGTERM, and SIGHUP (unix). Returns ShutdownReason enum (Signal/Fenced) without calling deregister — callers control deregistration timing. Refactored all 4 services (OMS, GW, RTMD-GW, Engine) to use the shared helper, eliminating duplicated signal handling. Engine's local ShutdownReason/wait_for_shutdown replaced with re-exports from infra.
