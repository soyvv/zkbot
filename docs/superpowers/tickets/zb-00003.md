# zb-00003 OKX gateway balance and position updates need exch_account_id linkage

- Ticket: zb-00003
- Date: 2026-03-24
- Time: 23:20 +08
- Domain: gateway, okx, oms, config
- Reporter: agent
- Status: resolved

## Description

OKX gateway balance and position updates are currently published without `exch_account_code`, so OMS cannot map exchange-owned state onto the internally managed account created in `cfg.account` and `cfg.account_binding`. The current GW semantic pipeline also populates internal `account_id` in these update messages, which is the wrong ownership boundary. For real venues such as OKX, the gateway should carry venue/account identity via exchange account code, not internal OMS account ID.

Need to add an OKX GW config field for `exch_account_id` and ensure the gateway encodes that value into outbound balance and position updates. OMS should continue linking inbound exchange-owned state through `exch_account_code`.

## Symptoms

- OMS logs warnings such as `unknown exchange account in balance update exch_account_code=""` when consuming OKX GW balance updates.
- OKX GW semantic pipeline publishes `BalanceUpdate` / `PositionReport` entries with empty `exch_account_code`.
- OKX GW semantic pipeline currently fills internal `account_id` into exchange-owned update messages.
- OMS balance/position routing logic expects `exch_account_code` on inbound exchange-owned state and drops updates when it is empty or unknown.

## Impact

- OKX balance updates do not link to the intended OMS-managed account.
- OKX position updates will have the same linkage failure mode.
- OMS/GW boundary is muddled: gateway events currently leak internal account identity instead of venue-native identity.
- The current design makes multi-account-per-gateway support harder because internal account IDs are not valid venue-side routing keys.

## Proposed Options Or Fixes

1. Add `exch_account_id` to OKX GW venue config schema and manifest descriptors so Pilot can store and bootstrap it.
2. Extend the OKX GW runtime config/adaptor to read `exch_account_id` and propagate it into all exchange-owned balance and position publications.
3. Update `zk-gw-svc` semantic pipeline so exchange-owned balance/position updates set `exch_account_code` and do not rely on internal `account_id` for OMS linkage.
4. Keep OMS linkage behavior as-is for exchange-owned state: resolve managed account through `cfg.account.exch_account_id` and `cfg.account_binding`.
5. Add regression coverage for OKX GW balance/position updates so OMS receives non-empty `exch_account_code` and links the updates to the correct account.

## Open Questions

- Should the generic GW semantic pipeline support both `account_id` and `exch_account_code`, with venue-specific policy deciding which one is authoritative?
- Should OMS explicitly reject inbound exchange-owned updates that carry internal `account_id` but no `exch_account_code`, to make the boundary violation more obvious?
- For venues with a single account per gateway, should `exch_account_id` still be mandatory in config, or can it be derived from secret material or account bootstrap metadata?

## Resolution

- Resolved At: 2026-03-25 20:34 +08
- Summary: Added exch_account_id to GwProvidedConfig/GwRuntimeConfig (ZK_EXCH_ACCOUNT_ID env var for direct mode). SemanticPipeline now sets exch_account_code (not internal account_id) on all outbound BalanceUpdate/PositionReport messages. OMS can now resolve exchange-owned state to the managed account via exch_account_code. OKX manifest updated with requires_exch_account_id metadata.
