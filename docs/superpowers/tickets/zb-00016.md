## Description

The Trading Console account table still loads slowly because /v1/accounts?include=summary,binding computes live per-account summary stats through OMS and gateway calls instead of using a cheap cached or deferred path.

## Symptoms

- Trading Console still shows long /accounts?include=summary,binding requests after the tab-level polling fix
- Account list refresh triggers per-account computeSummaryStats work even before the operator opens a detail tab
- computeSummaryStats calls queryPosition with queryGw=true and queryOpenOrders for every listed account
- UI can feel stuck while the top account table waits for live summary counts

## Impact

The main trading account page remains operator-visible slow even after hidden-tab polling was reduced. One table refresh becomes 1 + 2N live backend calls for N accounts, increasing OMS/GW load and obscuring real latency sources.

## Proposed Options Or Fixes

1. Split binding from summary so the account table can load quickly and hydrate summary later
2. Replace live summary fan-out with a cached or recorder-backed snapshot path for account counts
3. Remove queryGw=true from the list-summary path and reserve live gateway queries for focused detail panels or manual refresh
4. Make manual-ticket metadata and other secondary page metadata lazy so the main table path stays responsive

## Open Questions

- Should the account list show stale summary counts if that keeps the main table responsive?
- Should summary counts move to a separate lightweight API instead of remaining inside /v1/accounts?include=summary,binding?

## Resolution

- Resolved At: 2026-03-28 22:15 +08
