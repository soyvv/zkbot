# zb-00033 Introduce pluggable universe resolvers and implement Nasdaq Trader stocks/ETF resolver for IBKR refdata

## Description

IBKR refdata should not own hard-coded discovery logic for every possible universe source. Add a pluggable universe resolver architecture that turns source-specific candidate data into normalized IBKR contract specification strings, then reuses the existing IBKR reqContractDetails qualification/enrichment path. The first production-grade resolver should target US stocks and ETFs from the official Nasdaq Trader symbol directory files, covering nasdaqlisted.txt and otherlisted.txt with filters for ETF flag, exchange, test issues, and symbol/class normalization.

## Symptoms

- zb-00032 identifies that IBKR refdata currently has only an explicit universe list and no resolver layer
- IbkrRefdataConfig only accepts universe: tuple[str, ...] and ignores unknown future resolver fields
- refdata_config.schema.json only documents universe as a raw array of IBKR contract specification strings
- IbkrRefdataLoader.load_instruments returns [] for empty universe and otherwise directly qualifies each configured string through ContractTranslator.qualify and ib.reqContractDetailsAsync
- Operators need reproducible broad universes such as US ETFs or US stocks, but IBKR does not provide a bulk enumerate-all-symbols API
- Yahoo/yfinance is unsuitable as the canonical production universe source due to personal/research-use constraints and unofficial API stability; official listing files are a better first source

## Impact

Without a pluggable resolver boundary, each new universe source will be bolted directly into the IBKR loader and Pilot config. That makes it harder to test, hard to reuse across venues, and easy to confuse candidate discovery with IBKR qualification. A Nasdaq Trader resolver gives a practical, free, reproducible path for US stocks and ETFs while keeping IBKR as the authoritative contract enrichment step.

## Proposed Options Or Fixes

1. Define a small UniverseResolver interface or protocol that accepts resolver config and returns normalized candidate specs plus provenance metadata, e.g. AAPL-USD-STK-SMART
2. Add a universe_source discriminated schema with at least explicit and nasdaq_symbol_directory source types; keep legacy universe as shorthand for explicit
3. Implement NasdaqTraderSymbolDirectoryResolver that can read local files or configured URLs for nasdaqlisted.txt and otherlisted.txt, parse pipe-delimited symbol directory rows, filter ETF/test issues/exchanges, normalize symbols/classes for IBKR, and emit STK/USD/SMART contract specs
4. Use the existing IBKR ContractTranslator + reqContractDetails loop only after resolver output is built, preserving current enrichment fields such as minTick, multiplier, liquidHours, and contract identifiers
5. Add pacing controls around qualification using TokenBucketRateLimiter and config knobs such as max_candidates, max_qualifications_per_run, rate_limit_per_sec, and dry_run_candidates_only
6. Add tests with small checked-in fixtures for nasdaqlisted.txt and otherlisted.txt covering stock, ETF, test issue, class-share symbol normalization, duplicate handling, and exchange filters
7. Add operator/Pilot examples for US ETFs and US stocks, including clear warnings that Nasdaq directory gives listed candidate symbols, while IBKR qualification determines tradability/account entitlement

## Open Questions

- Should the pluggable resolver live inside venue-integrations/ibkr first, or under a shared refdata resolver package so other venues can reuse file/scanner/source resolvers?
- Should Nasdaq source support live HTTP download in the loader, or only local mounted files/artifacts to keep refdata refresh reproducible?
- What should be the canonical normalization for class-share symbols when mapping Nasdaq symbols to IBKR specs, for example BRK.B versus BRK B versus BRK-B?
- Should ETF filtering rely only on the Nasdaq Trader ETF flag, or should a later resolver enrich ETF metadata from another official/vendor source for AUM, issuer, leverage/inverse, expense ratio, etc.?
- Should empty resolver output fail the job, warn and return empty, or be configurable by strict mode?
