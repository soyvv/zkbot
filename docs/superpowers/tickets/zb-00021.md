# zb-00021 OMS does not run bounded gateway resync on late GW discovery, leaving state stale or incomplete

- Ticket: zb-00021
- Date: 2026-04-13
- Time: 21:46 +08
- Domain: oms, gw
- Reporter: codex
- Status: resolved

## Description

OMS performs startup balance, position, and order reconciliation only for gateways reachable during OMS startup. When a configured gateway registers later and OMS connects through discovery, OMS only adds the gRPC client and executor route; it does not run the same bounded resync. As a result, OMS can retain stale balances, positions, and order state after a late gateway connect or reconnect. In addition, the current order reconcile path only converges OMS-known open orders against gateway state and does not import gateway-only live orders into OMS.

## Symptoms

- OMS logs 'gateway registered in OMS from discovery' but no immediate balance/position/order reconcile follows
- A gateway that comes up after OMS startup can leave OMS snapshots stale until unrelated later events arrive
- Startup and periodic order reconcile operate on OMS-known open orders only, so gateway-only live orders are not materialized into OMS

## Impact

Late gateway availability or reconnect can leave OMS showing incorrect account state and incomplete order state, which is operationally dangerous after restarts or topology churn. The current reconnect path does not meet the documented intent that OMS should trigger a bounded resync on gateway restart or reconnect.

## Proposed Options Or Fixes

1. On successful GatewayDiscovered handling, trigger the same bounded resync used at OMS startup: query balances, query positions, run order reconcile, persist, and publish refreshed snapshots
2. Refactor startup reconcile logic into reusable writer-owned helpers so the same code path is invoked at startup, on late GatewayDiscovered, and optionally on explicit gateway reconnect events
3. Keep the immediate ticket scope to bounded late-connect resync for OMS-known state, and track gateway-only open-order import as a separate follow-up if you want to avoid changing OMS ownership semantics in the same patch
4. If full gateway-authoritative recovery is required, add a second-stage design for handling gateway-only open orders, for example by importing them as externally discovered orders or surfacing them as orphan venue orders for operator action

## Open Questions

- Should gateway-only open orders be imported into OMS automatically, or surfaced separately as orphan/external orders?
- Should balance and position resync also be triggered on repeated gateway reconnect/system-connected events, not only on first late discovery?
- Do we want a dedicated OmsCommand for gateway resync so discovery, reconnect, and manual admin triggers all share one path?

## Resolution

- Resolved At: 2026-04-13 23:30 +08
- Summary: Implemented bounded gateway resync on late GW discovery with shared reconcile helpers. Changes: (1) Moved normalize helpers to oms_actor.rs and passed confdata to writer loop. (2) run_gateway_discovered returns bool for first-discovery detection. (3) Extracted shared helpers: resync_gw_balance_position (balance + position query/persist) and reconcile_gw_orders (two-phase order reconcile). (4) All three reconcile paths (startup, late-discovery, periodic) now use the same shared helpers — consistent reconcile semantics everywhere. (5) Startup balance/position reconcile in main.rs replaced with call to shared helper. (6) Scope: OMS-known state convergence only — first discovery triggers full resync, repeated heartbeat upserts are ignored. (7) GW-only order import was attempted but reverted as unsafe (malformed order economics) — tracked as follow-up in zb-00023. Files: oms_actor.rs, reconcile.rs, main.rs. 12 reconcile tests + 55 total tests pass.
