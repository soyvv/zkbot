# zb-00037 Migrate refdata refresh to Prefect (or similar job runner)

- Ticket: zb-00037
- Date: 2026-04-27
- Time: 23:30 +08
- Domain: refdata
- Reporter: agent
- Status: deferred

## Description

User confirmed during planning of zb-0003x as a follow-up.

## Symptoms

- No retry on partial failures (e.g. transient IBKR Gateway disconnect mid-run): the run is marked 'failed' and the next periodic tick has to redo everything 5 min later.
- No backoff or alerting when consecutive runs fail.
- No way to schedule heavy refreshes off-peak (e.g. IBKR rebuild during US after-hours only).
- Refresh contention with the gRPC server process: a long-running refresh consumes the same event loop.

## Impact

Operational fragility scales linearly with universe size. Future addition of options chains or multi-currency stocks (~100k+ rows) will make the in-process loop untenable.

## Proposed Options Or Fixes

1. Move refresh_refdata into a Prefect flow with per-venue tasks; refdata-svc's coordinator becomes a thin wrapper that submits flows.
2. Or: run refdata-svc as 'reader-only' (just gRPC) and have a separate refdata-worker process drive Prefect flows that write to the same DB.
3. Or: simpler — keep the in-process loop but add a per-venue retry-with-backoff policy and emit Prometheus metrics on run duration / failure rate.

## Open Questions

- Does Prefect deployment infra exist in zkbot ops yet? If not, the 'simpler' option may be the right next step.
- Should refresh_sessions and refresh_refdata move together, or stagger?

## Resolution (deferred)

Resolved-by-deferral on 2026-04-29. The architectural Prefect migration is
explicitly out-of-scope for the current iteration. The MVP of the simpler
option (per the ticket's option 3) is already in place via:

- **zb-00042** circuit-breaker on socket disconnect — failed runs no longer
  log-spam; the next periodic tick retries cleanly.
- **zb-00036** registry decoupled from initial refresh — service stays
  reachable while a long refresh runs in the background.
- **zb-00035** loader cache + per-venue lock in RefreshCoordinator —
  refresh_sessions and refresh_refdata serialize properly without racing
  fresh connections.

At today's scale (~11k IBKR + ~1.5k OKX + ~127 OANDA rows) the in-process
asyncio loop with periodic retry works. Re-open this ticket when:
1. Universe sizes grow past ~50k rows per venue (options chains, multi-currency
   stocks), OR
2. We need cross-process job orchestration (off-peak scheduling, replay,
   structured retries beyond circuit-breaker), OR
3. The user explicitly asks for Prefect.
