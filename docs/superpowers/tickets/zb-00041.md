# zb-00041 useInstrumentSearch backend doesn't accept type/status filters; UI applies them client-side

- Ticket: zb-00041
- Date: 2026-04-27
- Time: 23:31 +08
- Domain: pilot
- Reporter: agent
- Status: resolved

## Description

The new search box on the Refdata page calls useInstrumentSearch(q, venue, limit) which hits Pilot's GET /v1/refdata/instruments with q+venue. type and status dropdown filters are then applied client-side over the bounded search result page, which is correct functionally but inconsistent with how the non-search browse path (useInstruments) sends them as backend query params.

## Symptoms

- Searching 'AAPL' with type=stock filter applied: backend returns all matches for 'AAPL' irrespective of type, then UI filters in JS.
- For a venue with many same-symbol matches across types, this could exceed the search backend's bounded result page (e.g. limit=500) before client filtering kicks in.

## Impact

Edge case today (most venues have unique symbols per type), but a real correctness gap as venue universes grow. Inconsistent behavior between search-mode and browse-mode dropdowns may surprise operators.

## Proposed Options Or Fixes

1. Add type and status as accepted query params on the search backend path (Pilot RefdataController + RefdataService). UI hook signature becomes useInstrumentSearch(q, venue, type, status, limit).
2. Or: document the client-side filtering as intentional and add a test that the search result page is large enough not to truncate before filters apply.

## Open Questions

- Is the search backend currently a separate code path from the list backend, or do they share the same SQL? Looks like the same RefdataRepository.searchInstruments — adding the params is trivial.

## Resolution

- Resolved At: 2026-04-29 00:19 +08
- Summary: Backend RefdataController.searchInstruments + RefdataRepository now accept 'status' query param (LOWER(lifecycle_status) = LOWER(?), case-insensitive). UI hook useInstrumentSearch extended to take (q, venue, type, status, limit) and forwards all filters to backend. RefdataPage callsite passes typeFilter+statusFilter through. Removed redundant client-side filter (instrumentsBase + useMemo) — both search and browse now hit the same backend filters uniformly. queryKey includes type/status so cache invalidates correctly. TypeScript compile clean. Search backend (case-insensitive UPPER on type, LOWER on status) matches DB conventions: instrument_type stored UPPERCASE per zb-00043 normalization, lifecycle_status stored lowercase.
