# zb-00019 Unify create/edit definition forms across Pilot domains

- Ticket: zb-00019
- Date: 2026-04-11
- Time: 18:38 +08
- Domain: pilot, design
- Reporter: Codex
- Status: resolved

## Description

Pilot create and edit flows for services, accounts, bots, strategies, and refdata do not share a domain-level form contract. They reuse low-level field rendering in places, but each page independently composes identity fields, schema-driven config, linked entities, mutability, and payload building, causing create/edit drift and inconsistent operator workflows.

## Symptoms

- GW service onboarding and service edit render different layouts for the same underlying config schema.
- Service onboarding includes identity, config, and bindings in one workflow, while service edit only edits providedConfig inside ServiceDetailPage.
- Account create uses a modal for account master data while account edit only changes OMS/GW binding in AccountDetailPage, so there is no single account definition form.
- Bots use one routed page for create/edit but still branch mode-specific composition and submission logic.
- Strategies use inline create/edit forms in StrategiesPage instead of a shared routed or management-panel definition form.

## Impact

Operators cannot rely on one learned form structure across create and edit, increasing onboarding friction and making schema-driven changes risky. UI regressions recur because parity depends on multiple page-specific implementations staying in sync. The current composition also hides domain immutability rules in ad hoc UI code instead of one explicit field-policy model.

## Proposed Options Or Fixes

1. Introduce one domain-level definition form component per entity type, for example ServiceDefinitionForm, AccountDefinitionForm, BotDefinitionForm, and StrategyDefinitionForm, each used by both create and edit wrappers.
2. Define explicit field mutability policies such as create_only, editable, derived, and runtime_only, and centralize default-value hydration and payload building.
3. Separate definition editing from runtime-only operational editing, especially for accounts and services, so create/edit symmetry applies to the correct resource boundary.
4. Replace CreateAccountDialog with a routed account definition page and use the same form shell for account edit, with immutable identifiers rendered read-only rather than omitted.
5. Move strategy create/edit off the current inline table editor toward a routed page or dedicated management-panel workflow, consistent with UI consistency guidance.

## Proposed Implementation Plan

1. Introduce shared definition-form primitives under `zkbot-ui/pilot-console/src/components/forms/definition/` and `zkbot-ui/pilot-console/src/lib/forms/`.
   Add common concepts for:
   - `FormMode = "create" | "edit"`
   - field mutability policy such as `create_only`, `editable`, `read_only`, `derived`, and `runtime_only`
   - default-value hydration from API entities
   - payload builders for create and update
   - schema section splitting for common and venue-specific config

2. Refactor services first.
   Extract a `ServiceDefinitionForm` from the current onboarding flow and reuse it from service edit.
   Target structure:
   - `Identity`
   - `Configuration`
   - `Venue Configuration` when applicable
   - `Bindings`
   Use the same layout and widgets in create and edit, with explicit field-level mutability.

3. Refactor accounts second.
   Replace `CreateAccountDialog` with a routed `CreateAccountPage` and introduce `AccountDefinitionForm`.
   Decide and encode whether OMS/GW binding is:
   - part of the account definition form, or
   - a separate operational section with a separate save action
   Show immutable identifiers such as `accountId` read-only in edit mode instead of removing them.

4. Refactor strategies third.
   Extract `StrategyDefinitionForm` from the current inline create and edit implementations.
   Move strategy create and edit to either:
   - a routed page, or
   - a persistent lower management panel
   Remove duplicate page-local payload and validation handling.

5. Refactor bots fourth.
   Extract `BotDefinitionForm` from `OnboardBotPage`.
   Keep the routed workflow, but reduce mode-conditional branching in the page and centralize config submission rules in the shared form.

6. Refactor refdata venue onboarding and future edit flows last.
   Align refdata with the same definition-form shell so venue-backed configuration follows the same composition rules as services.

## Acceptance Criteria

- For a given entity type, create and edit render the same section structure and field ordering.
- Immutable fields remain visible in edit mode as read-only fields unless there is a documented reason to hide them.
- Schema-driven config uses one shared composition path for create and edit, including venue-specific nested config.
- Create and edit no longer duplicate page-local default-value mapping and payload-building logic.
- Account create no longer uses a standalone modal if account edit is page-based.
- Strategy create and edit no longer rely on separate inline implementations inside `StrategiesPage`.
- Runtime-only fields and definition fields are explicitly separated in code and in the UI.
- UI differences between create and edit are limited to:
  - field mutability
  - action labels
  - create-only follow-up actions such as bootstrap token issuance

## Migration Notes

- Do not begin by copying existing create JSX into edit pages.
- Start with extracting shared domain forms from the more complete create workflows, then embed them into edit flows.
- Prefer preserving backend contracts first and isolating UI refactors unless a definition-versus-runtime boundary decision requires API work.
- If account binding ownership is unresolved, document the chosen boundary before implementing `AccountDefinitionForm`.

## Open Questions

- For accounts, should OMS/GW binding be part of the account definition form or remain a separate operational section with its own save action?
- Which service fields are intentionally immutable after creation, for example logicalId, venue, or exchAccountId-style external identifiers?
- Should service edit expose the full onboarding workflow including bindings and enabled state, or should some sections remain separate operational actions?
- Do bots and strategies need a single routed create/edit workspace as the standard for all schema-driven definition forms?

## Resolution

- Resolved At: 2026-04-11 19:13 +08
- Updated At: 2026-04-11
- Summary: Phase 5 complete for services, strategies, and bots. Post-implementation fixes applied for venue fallback and empty-config edit semantics.

### What was unified
- **Services:** ServiceDefinitionForm handles both create (OnboardServicePage) and edit (ServiceDetailPage). Split venue schema rendering is consistent. Venue field loss bug fixed via effectiveVenue fallback from existingConfig.
- **Strategies:** StrategyDefinitionForm + routed StrategyFormPage (/strategies/create, /strategies/$strategyId/edit). StrategiesPage reduced to table-only.
- **Bots:** BotDefinitionForm extracted from OnboardBotPage. Config rendering uses shared SchemaConfigSection.
- **Shared infra:** SchemaConfigSection (5-state rendering), resolveConfig with mode-aware empty handling, readiness helpers.

### Account scope decision
Account edit is intentionally binding-only (omsId, gwId via UpdateAccountRequest). Account identity fields (accountId, venue, brokerType, etc.) are immutable after creation — there is no definition-level edit. The ticket's open question "should OMS/GW binding be part of the account definition form?" is resolved as **no** — binding stays as a separate operational section on AccountDetailPage. Account create was moved from modal (CreateAccountDialog) to routed page (/trading/create) per acceptance criteria.

### Post-implementation fixes (2026-04-11)
1. **Venue fallback:** ServiceDefinitionForm now extracts venue from existingConfig.venue when the prop venue is undefined (backend may return detail.venue: null while providedConfig.venue is set).
2. **Empty config clear:** resolveConfig() now accepts a mode parameter. In edit mode, empty config returns {} instead of undefined, so the server receives the explicit clear intent. Strategy and bot edit forms always include config in the update payload.
