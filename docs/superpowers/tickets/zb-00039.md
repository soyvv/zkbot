# zb-00039 ExplicitResolver hardcodes is_etf=False; ETF tickers in operator-curated lists misclassified as 'stock'

- Ticket: zb-00039
- Date: 2026-04-27
- Time: 23:31 +08
- Domain: refdata
- Reporter: agent
- Status: resolved

## Description

ExplicitResolver wraps every item in the operator-supplied 'universe' list as ResolvedInstrument(spec=item, is_etf=False). When operators hand-author an ETF watch-list (e.g. universe: ['SPY-USD-STK-SMART', 'QQQ-USD-STK-SMART']), every row lands in cfg.instrument_refdata with instrument_type='stock' even though they are ETFs.

## Symptoms

- Operator authors universe: ['SPY-USD-STK-SMART'].
- After refresh: cfg.instrument_refdata row for SPY has instrument_type='stock', asset_class='stock'.
- Strategies that filter by instrument_type='etf' will not see SPY.

## Impact

Operator-side ETF filters silently drop legitimate ETFs from explicit lists. Codex flagged this as NIT during review of zb-0003x; user accepted the simpler default but the correctness gap remains for explicit-list users.

## Proposed Options Or Fixes

1. Accept richer config item shape: explicit.config.items can be a list of strings (default is_etf=False) OR a list of {spec, is_etf} objects. Document in refdata_config.schema.json.
2. Or: introduce a separate 'explicit_etfs' resolver type whose items are all is_etf=True.
3. Or: detect ETF heuristically by symbol pattern (-, .V, leveraged 2x/3x, etc.) — fragile, not recommended.

## Open Questions

- Is there a real operator workflow where someone hand-authors a small ETF list (e.g. for a specific strategy) instead of using the Nasdaq directory? If yes, option (a) is the right fix; if no, ignore.

## Resolution

- Resolved At: 2026-04-29 00:16 +08
- Summary: Extended ExplicitResolver to accept dict items {spec, instrument_type} alongside plain strings. instrument_type accepts canonical UPPERCASE proto names (ETF, STOCK, FUTURE, ...), case-insensitive lowercase ('etf'), or proto int values. Validation happens at construction time so operator config errors surface eagerly. is_etf flag-pair was already removed in Item 7c (replaced with proto-aligned instrument_type field on ResolvedInstrument). Operators can now write: items: [{spec: 'SPY-USD-STK-SMART', instrument_type: 'ETF'}, ...]. Bare-string items continue to work and default to UNSPECIFIED (loader falls back to STK→STOCK).
