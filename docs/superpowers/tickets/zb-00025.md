# zb-00025 OANDA positions are misclassified as spot and do not refresh live in OMS

- Ticket: zb-00025
- Date: 2026-04-14
- Time: 00:09 +08
- Domain: gw, oms, refdata
- Reporter: codex
- Status: resolved

## Description

OANDA gateway positions are not represented correctly in OMS because OANDA position facts use instrument strings like BTC_USD/EUR_USD and the GW host infers them as spot. OMS therefore routes them through the balance path instead of the exchange-position path. Separately, the OANDA adaptor does not emit live position events, so post-startup position state does not refresh from runtime activity.

## Symptoms

- OMS startup reconcile logs show position reconcile done for gw_oanda_demo1, but Pilot still shows incorrect or missing OANDA positions
- Current OANDA GW log shows balance publication for gw_oanda_demo1 but no gateway position publication lines
- OANDA normalize_positions emits instruments like EUR_USD and BTC_USD, which do not satisfy the current non-spot heuristics in zk-gw-svc
- OMS classifies spot-type gateway entries as balances and only non-spot entries become exchange positions

## Impact

OANDA exchange positions can be absent or stale in OMS and Pilot, even when the venue reports them. This breaks operator visibility, weakens reconcile correctness, and risks downstream logic depending on exchange position state. The issue also means startup query_position reconcile can silently ingest OANDA positions into the wrong classifier, not just the live streaming path.

## Proposed Options Or Fixes

1. Add explicit instrument_type to VenuePositionFact and have venue adaptors provide authoritative position classification instead of host-side string heuristics
2. Update the OANDA adaptor to mark OANDA CFD positions as non-spot/derivative and add targeted tests covering BTC_USD/EUR_USD classification through both live event and query_position paths
3. Add a live OANDA position publication path so fills/reconcile cycles emit VenueEvent::Position snapshots, not only balances
4. As a short-term fallback, special-case OANDA underscore CFD symbols in zk-gw-svc heuristics, but treat that only as a temporary bridge until explicit type propagation is in place

## Open Questions

- Should OANDA positions be modeled as perp-like or as a dedicated CFD classification within existing instrument_type semantics
- Do we want live OANDA position refresh on every fill event, periodic reconcile tick, or both
- Are current OANDA refdata / metadata rows keyed by venue exchange symbols that can be resolved directly by gw_id, or do we need a symbol normalization layer as part of this fix

## Resolution

- Resolved At: 2026-04-14 00:29 +08
- Summary: Fixed OANDA positions misclassified as spot and missing live position events. Added instrument_type field to VenuePositionFact with serde default for backwards compat. semantic_pipeline and grpc_handler use fact instrument_type when non-zero, falling back to string heuristic. OANDA normalize_positions sets instrument_type=4 (CFD). OANDA adapter now emits position events at connect and during periodic reconcile.
