# zb-00020 OANDA GW Python event loop exits on idle timeout causing lost OMS linkage and ineffective reconciliation

- Ticket: zb-00020
- Date: 2026-04-12
- Time: 23:58 +08
- Domain: gw, oms
- Reporter: codex
- Status: resolved

## Description

The OANDA Python gateway publisher path stops consuming venue events after an idle next_event timeout, so OMS misses linkage and later state or cancel reports. OMS reconciliation then rejects pending orders with no exch_order_ref instead of recovering them.

## Symptoms

- GW log shows event loop exiting on python next_event TimeoutError after idle period
- Orders accepted by GW remain pending in OMS with no linkage report
- OMS periodically logs reconcile: pending order has no exch_order_ref, rejecting for OANDA orders
- OANDA adaptor compensating logic enqueues reconcile events but they never reach OMS once the GW event loop has exited

## Impact

OANDA order lifecycle becomes unreliable after the gateway has been idle long enough to hit the Python next_event timeout. OMS can neither receive normal linkage and cancel reports nor recover non-linked pending orders after restart or drift, leaving orders stuck pending or incorrectly rejected.

## Proposed Options Or Fixes

1. Make Python next_event idle timeout non-fatal in zk-pyo3-bridge so the GW event publisher loop keeps running across idle periods
2. Add a targeted GW and Python bridge test that verifies long idle periods do not terminate the publisher task
3. Keep OANDA compensating logic, but treat it as secondary to a resilient event-consumer loop
4. Improve OMS reconciliation for pending orders without exch_order_ref so it can attempt a recoverable lookup path instead of immediate rejection when venue semantics allow it

## Open Questions

- Should next_event block indefinitely for Python adaptors, or should the bridge convert idle timeout into a retryable no-event condition?
- Do we want OMS reconcile to support client-order-id-based recovery for venues like OANDA, or keep exch_order_ref as the only recovery handle?

## Resolution

- Resolved At: 2026-04-13 21:15 +08
- Summary: Fixed in two layers: (1) Bridge: next_event() in gw_adapter.rs and rtmd_adapter.rs now wraps the timeout+spawn_blocking call in an internal retry loop. Only the specific 'next_event: TimeoutError' from Future.result(timeout) is retried — other PythonCall/TransientError forms are propagated as real failures. No changes to py_errors.rs or service main.rs files. (2) OANDA adaptor: next_event() now uses wait_for(queue.get(), 30s) with a _check_stream_health() watchdog that monitors last_activity timestamp (not task.done()), detects stale streams, and restarts with a fresh OandaTransactionStream object. Tests: 14 Rust bridge tests pass (including new idle-timeout integration test), 6 Python health watchdog tests pass. Both services build with --features python-venue.
