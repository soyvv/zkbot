# zb-00005 Implement canonical OMS position query and publication semantics

- Ticket: zb-00005
- Date: 2026-03-25
- Time: 23:55 +08
- Domain: oms, query, eventing, runtime
- Reporter: agent
- Status: open

## Description

The refactor design requires one canonical OMS position view:

- executable quantity comes from OMS-managed positions
- exchange-owned economics and metadata come from exchange position snapshots when available

Current OMS behavior still exposes a split model:

- `query_gw=true` returns exchange snapshots
- default `QueryPosition` returns managed positions only
- `PositionUpdateEvent` is built from managed positions only

Need to implement the merged canonical position view and make query/event semantics consistent.

## Symptoms

- `QueryPosition` has two disjoint modes rather than one canonical merged result
- `PositionUpdateEvent` does not include exchange-owned economics
- runtime readers and event consumers do not observe the same position contract

## Impact

- the live OMS API does not match the design doc
- downstream consumers must guess which path contains the authoritative position picture
- reconcile state cannot be surfaced cleanly on one canonical model

## Proposed Options Or Fixes

1. Define a merged canonical position materialization function in `zk-oms-rs`.
2. Rewrite `QueryPosition` to use that canonical merged view.
3. Rewrite `PositionUpdateEvent` building to publish the same canonical merged view.
4. Add comments/tests documenting precedence rules:
   - managed quantity wins for executable fields
   - exchange snapshot wins for exchange-owned economics
5. Add regression tests covering:
   - managed-only position
   - exchange-only position
   - merged managed + exchange position

## Acceptance Criteria

- `QueryPosition` returns one canonical merged view by default.
- `PositionUpdateEvent` matches the canonical query semantics.
- Tests lock the merge precedence between managed and exchange-owned fields.
