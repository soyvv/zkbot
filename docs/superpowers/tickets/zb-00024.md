# zb-00024 Python-backed gateway adaptors lack explicit shutdown lifecycle, causing stuck or slow Ctrl+C termination

- Ticket: zb-00024
- Date: 2026-04-13
- Time: 22:53 +08
- Domain: gw, design
- Reporter: codex
- Status: resolved

## Description

Python-backed gateways such as OANDA can remain slow or stuck on Ctrl+C because the Rust gateway host detects shutdown but does not explicitly stop the spawned adapter event-loop task or invoke adaptor shutdown. The Python bridge runs next_event() through spawn_blocking plus run_coroutine_threadsafe, and Python venue adaptors maintain their own asyncio background tasks and HTTP/stream resources. Without a host-level shutdown hook, those resources may outlive the main shutdown path and delay process exit. This appears to be a common issue for Python-backed adaptors, not OANDA-specific.

## Symptoms

- OANDA GW may not stop promptly on Ctrl+C even though the Rust service has a shutdown signal path
- zk-gw-svc spawns a background adapter event task and does not retain or abort the JoinHandle during shutdown
- Python venue adaptors such as OANDA implement shutdown() logic, but the Rust VenueAdapter contract does not expose a lifecycle shutdown method for the host to call

## Impact

Gateway processes can hang or exit slowly during operator shutdown, local dev iteration, and service restarts. Stale processes also delay KV deregistration and can confuse Pilot and operational tooling. Because the issue sits in the Python bridge and host lifecycle boundary, other Python-backed GW or RTMD adaptors are likely affected too.

## Proposed Options Or Fixes

1. Extend the VenueAdapter lifecycle contract with an async shutdown or disconnect method and implement it for both Rust-native and Python-backed adaptors
2. In zk-gw-svc, retain the spawned adapter event-loop JoinHandle and explicitly abort or stop it during shutdown before process exit
3. For Python-backed adaptors, have the bridge call Python shutdown() when present and fall back to disconnect() or a no-op when not implemented
4. Review the Python bridge next_event implementation and outstanding run_coroutine_threadsafe futures so shutdown can cancel or drain them cleanly instead of relying on runtime teardown

## Open Questions

- Should shutdown be added to the shared VenueAdapter trait as shutdown(), disconnect(), or a separate optional lifecycle capability?
- Do RTMD Python adaptors need the same host-side shutdown treatment immediately, given they use the same bridge pattern?
- Should the gateway host stop the event-consumer task before calling adaptor shutdown, or call adaptor shutdown first to unblock next_event?

## Resolution

- Resolved At: 2026-04-13 23:02 +08
- Summary: Added shutdown() to VenueAdapter and RtmdVenueAdapter traits (default no-op). PyEventLoop now has explicit shutdown(&self, timeout) that stops the asyncio loop and joins the background thread with timeout (Mutex<Option<JoinHandle>> for &self access through Arc). PyVenueAdapter and PyRtmdVenueAdapter implement shutdown() by calling Python adapter shutdown() if present, then shutting down the event loop. Both zk-gw-svc and zk-rtmd-gw-svc now capture the event-loop task JoinHandle, call adapter.shutdown() during graceful exit, and abort the event-loop consumer task. This ensures Python adapters like OANDA get their shutdown() called (cancelling streams, closing HTTP clients) and the event loop thread is joined cleanly on SIGINT/SIGTERM/SIGHUP.
