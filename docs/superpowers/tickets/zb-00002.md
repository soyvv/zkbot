# zb-00002 PUT /v1/accounts/{accountId} does not support account binding updates

- Ticket: zb-00002
- Date: 2026-03-24
- Time: 21:09 +08
- Domain: pilot, db
- Reporter: agent
- Status: resolved

## Description

Frontend AccountDetailPage sends { omsId, gwId } in PUT /v1/accounts/{accountId} body but backend ignores these fields. The design doc (pilot_ui_interaction.md Sequence 1A) specifies binding updates via this endpoint but the backend UpdateAccountRequest DTO only has a status field. Need to add a write path for cfg.account_binding from the account update API.

## Symptoms

- Frontend AccountDetailPage sends `{ omsId, gwId }` in PUT body but backend ignores them.
- `UpdateAccountRequest` Java DTO only has `status` field.
- `AccountRepository` has `getAccountBinding()` but no upsert method.
- `cfg.account_binding` table exists with `(account_id, oms_id, gw_id)` composite PK but no API write path.

## Impact

Account binding editing UI is non-functional. Operators cannot reassign accounts to different OMS/GW instances through Pilot UI.

## Proposed Options Or Fixes

1. **Option A**: Extend `AccountService.updateAccount()` to accept `omsId`/`gwId` in `UpdateAccountRequest`. Add `AccountRepository.upsertAccountBinding()`. Enforce offline-only rule (reject if OMS is live in DiscoveryCache).
2. **Option B**: Dedicated endpoint `PUT /v1/accounts/{accountId}/binding` with a separate `BindingUpdateRequest` DTO. Keeps status and binding updates on separate paths.
3. **Option C**: Reuse `PUT /v1/topology/bindings` with `srcType=ACCOUNT`. Requires two frontend calls and uses `cfg.logical_binding` instead of `cfg.account_binding`.

## Open Questions

- Should backend validate that target OMS/GW instances exist before accepting the binding update?
- Should binding updates cascade-invalidate related runtime state (e.g. mark account as needs-resync)?
- Does the offline-only enforcement check OMS liveness via DiscoveryCache or via session table?
