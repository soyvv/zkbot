# zb-00038 AG Grid server-side row model migration for Refdata Instruments tab

## Description

The Refdata page currently fetches up to 5,000 rows per venue and paginates client-side via AG Grid's basic pagination. The Pilot backend caps at 20,000 rows. This is a temporary band-aid; once a venue grows past ~20k rows (or when multiple venues need to be browsed together), the eager-fetch model breaks.

## Symptoms

- Selecting IBKR fetches all ~11k rows up front (~few MB JSON) on every page load.
- No support for combined-venue browse without raising the cap further.
- Server-side limit is hardcoded at Math.min(limit, 20000) in RefdataRepository.java with a TODO comment but no migration path.

## Impact

Bounded by the temporary ceiling. When a second venue (e.g. multi-currency CFDs, options chains) grows past 20k, operators will not see all instruments. Initial page load latency grows with universe size.

## Proposed Options Or Fixes

1. Adopt AG Grid Enterprise's server-side row model (paid feature) — Pilot exposes a paginated/sorted/filtered REST endpoint and the grid lazy-loads pages on scroll.
2. Or: stay community-tier but add cursor-based pagination + a sticky 'Load more' button at the table footer.
3. Or: keep eager fetch but compress payload (only ship BASIC disclosure fields by default; detail-on-click for EXTENDED/FULL).
4. Search box (Item 3 of zb-0003x) already covers the 'find a specific symbol' case, so server-side grid is mostly about browse / sort / filter UX.

## Open Questions

- Does the team have an AG Grid Enterprise license, or should we stay community-tier?
- What's the expected upper bound on instrument count per venue in 2026 — 50k, 200k, more?

## Resolution (deferred)

Resolved-by-deferral on 2026-04-29. The current temporary ceiling (UI sends
limit=5000, backend caps at 20000, page-size selector exposes [20,50,100,200,500])
is sufficient for today's largest venue (IBKR ~11k rows). Migrating AG Grid
to server-side row model requires either an Enterprise license or a
custom-paginated REST scheme — both larger than the immediate need.

zb-00041 added backend-side type+status filtering to the search hook so the
UI can narrow large result sets via filters instead of paging through all
rows. zb-00033's Nasdaq resolver supports max_candidates capping.

Re-open when:
1. A second venue grows past ~20k rows (or IBKR universe expands to options
   chains / multi-currency), making the 20k ceiling actually painful, OR
2. The team decides to license AG Grid Enterprise, OR
3. Operators report meaningful degradation when browsing the full instrument
   list across venues.
