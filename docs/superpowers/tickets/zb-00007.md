# zb-00007 Implement OMS pending-order restart reconcile and periodic order recheck

## Description

OMS V2 does not currently self-heal warm-started pending/live orders after restart.

On restart, OMS reloads live orders from Redis, but startup reconciliation only queries
gateway account balances. There is no startup `QueryOrder` or `QueryOpenOrders` sweep for
warm-started orders, and there is no active periodic order recheck path in the live V2
runtime.

As a result, if a gateway reject or terminal order update is missed while OMS is down, the
order can remain stuck in `Pending` after restart.

## Symptoms

- pending/live orders are warm-started from Redis on OMS restart
- startup reconcile only calls gateway `query_account_balance`
- `RecheckOrder` and `RecheckCancel` are defined but dropped by `OmsCoreV2`
- the only periodic recheck timer is `PositionRecheck`, and that path is still a stub
- gateway report subscribers use plain NATS at-most-once delivery, so missed terminal
  reports are not replayed durably after OMS restart

## Impact

- OMS can show stale pending orders indefinitely after restart
- operators must manually inspect venue state to determine whether an order actually failed,
  filled, or was cancelled
- OMS state convergence depends too heavily on uninterrupted live report delivery
- restart recovery does not meet the intended hybrid warm-start plus gateway reconcile model

## Proposed Options Or Fixes

1. Add startup order reconcile for warm-started non-terminal orders using gateway
   `QueryOrder` and/or `QueryOpenOrders`.
2. Add a periodic order recheck timer in `zk-oms-svc`, separate from position recheck.
3. Port `RecheckOrder` and `RecheckCancel` behavior into `OmsCoreV2`, or move the full
   logic into the service layer with explicit gateway query and convergence actions.
4. Define convergence rules for restart reconcile:
   - live on exchange: update OMS with current order state
   - missing on exchange and no exchange ref: synthetically reject stale pending order
   - missing on exchange with known exchange ref: terminalize according to venue/query result
5. Add regression tests covering a warm-started pending order that becomes terminal after
   startup requery.

## Acceptance Criteria

- OMS restart actively reconciles warm-started non-terminal orders against gateway state.
- A stale pending order converges to a terminal state after startup reconcile when the
  gateway no longer reports it as live.
- OMS has an active periodic order recheck path for pending/live orders.
- The runtime no longer relies solely on best-effort live NATS report delivery to clear
  stale pending orders.

## Resolution

- Resolved At: 2026-03-25 23:46 +08
- Summary: Implemented OMS order reconciliation (startup + periodic). Startup: extracts open order snapshots, queries each gateway with 10s timeout, feeds synthetic OrderReports through core.process_message. Periodic: OrderReconcile command on order_resync_interval_secs (60s default). Convergence: live orders updated on fill divergence, missing orders with exch_ref terminated, pending orders without exch_ref rejected. 6 unit tests. Unstubbed gateway query_order_details/query_open_orders, implemented simulator query support.
