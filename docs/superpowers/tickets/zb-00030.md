# zb-00030 Mark superseded runtime-contract docs as historical and align remaining operator docs with the current package-based structure

- Ticket: zb-00030
- Date: 2026-04-19
- Time: 13:05 +08
- Domain: design, strategy, gw, mdgw, pilot
- Reporter: agent
- Status: open

## Description

The repo's active code structure has largely moved to top-level python/, java/, rust/, and venue-integrations/ with explicit dependency-contract rules, but repo search still surfaces stale design specs, onboarding docs, and operator-facing references that describe removed path-based behaviors such as ZK_VENUE_ROOT, python_search_path, ZK_ENGINE_BINARY_PATH, sys.path injection, or maturin develop. This creates architectural ambiguity even when the active code is moving in the right direction.

## Symptoms

- venue-integrations/README.md and navigation docs drifted from the active tree and needed direct correction during this review
- historical specs under docs/superpowers/specs and system-redesign-plan still describe ZK_VENUE_ROOT and sys.path-based loading as if they were current design guidance
- current user-facing docs such as docs/strategies/ETS/README.md and docs/system-arch/services/engine_service.md still contain python_search_path guidance while dependency-contract.md forbids it
- Pilot config docs and code-search results still surface ZK_ENGINE_BINARY_PATH references, which conflicts with the newer PILOT_ENGINE_BINARY_PATH contract
- searching the repo for venue/runtime guidance returns a mix of canonical current docs and superseded plans/specs, making it hard for operators and agents to know which contract is authoritative

## Impact

This slows onboarding and code review because the repository presents multiple contradictory runtime models at once. Agents and humans can easily follow a stale spec instead of the current contract, especially when using grep-based discovery. The result is repeated confusion about package boundaries, startup env vars, and how venue modules are supposed to be loaded.

## Proposed Options Or Fixes

1. Audit all non-historical operator-facing docs and architecture docs for removed contract terms such as ZK_VENUE_ROOT, python_search_path, ZK_ENGINE_BINARY_PATH, sys.path injection, and maturin develop, then either update or delete them
2. Introduce a clear historical-doc convention for superseded plans and specs, for example a front-matter flag or a leading banner that marks the file as non-authoritative and points to dependency-contract.md / arch_design_decisions.md
3. Update repo-routing docs and skills so search-first workflows prioritize canonical docs under docs/system-arch and de-prioritize or explicitly label historical specs and redesign notes
4. Add a lightweight docs audit script or CI grep that checks active docs for removed runtime-contract terms, excluding explicitly historical directories

## Open Questions

- Which doc trees should count as authoritative versus historical for search and agent-routing purposes?
- Should superseded specs remain in place with banners, or be moved under an archive/ directory to reduce accidental discovery?
- Should user-facing onboarding docs such as venue-integrations/README.md and strategy READMEs be included in the same CI audit as system-arch docs?
