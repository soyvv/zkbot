# zb-00018 Pilot /v1/bot/definitions list path is slow and appears to scan latest execution per bot without a supporting index

- Ticket: zb-00018
- Date: 2026-03-28
- Time: 22:49 +08
- Domain: pilot
- Reporter: Codex
- Status: resolved

## Description

The Bots page can hang on GET /v1/bot/definitions. The current list query joins each engine_instance to a lateral subquery over cfg.strategy_instance to find the current or last run, but the schema only has a plain engine_id index and no composite index aligned with ORDER BY status-priority, started_at desc.

## Symptoms

- Browser network panel shows GET /v1/bot/definitions taking a very long time or staying pending on the Bots page
- BotDefinitionRepository.listBotDefinitions() uses a LEFT JOIN LATERAL subquery against cfg.strategy_instance for every bot row
- The lateral subquery orders by active-status priority and started_at desc, which is not covered by the current indexes
- As execution history grows, the bot list endpoint is likely to get slower even before any bot detail panels load

## Impact

The main Bots page can become operator-visible slow or appear frozen. Listing bot definitions is a primary navigation path, so a slow latest-run join degrades routine bot management and can block lifecycle actions from the UI.

## Proposed Options Or Fixes

1. Add a supporting composite index for latest-run lookup on cfg.strategy_instance, for example over (engine_id, started_at desc) and consider a partial/covering approach for active statuses
2. Materialize current_or_last_execution on engine_instance or in a dedicated summary table so the list path does not compute it per request
3. Split /v1/bot/definitions into a fast base list and a separate run-summary hydrate path if latest-run context is not needed on first paint
4. Profile the SQL with EXPLAIN ANALYZE against current local data to confirm whether the lateral join is the dominant cost

## Open Questions

- Should the Bots list show current or last run on initial load, or can that hydrate after the base definitions render?
- Is the intended steady-state model a live join against strategy_instance, or should Pilot persist bot run summary separately?

## Resolution

- Resolved At: 2026-03-28 22:52 +08
- Summary: Added composite index idx_strategy_instance_engine_started on cfg.strategy_instance(engine_id, started_at DESC) in 00_schema.sql. This covers the LATERAL subquery in listBotDefinitions() and getBotDefinition() which filters by engine_id and orders by started_at DESC. The old single-column engine_id index forced a full scan + sort per engine; the new index allows an index-only top-1 pick. For existing databases: CREATE INDEX CONCURRENTLY idx_strategy_instance_engine_started ON cfg.strategy_instance(engine_id, started_at DESC);
