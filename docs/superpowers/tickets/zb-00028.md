# zb-00028 Unify protobuf code generation into shared per-language generated libraries

- Ticket: zb-00028
- Date: 2026-04-18
- Time: 13:44 +08
- Domain: design, gw, mdgw, engine, strategy
- Reporter: agent
- Status: open

## Description

The repo currently duplicates generated Python protobuf code inside venue packages, generates BetterProto models into a separate shared library, and uses separate ad hoc generation paths for Python, Java, and Rust. We should refactor proto layout so schema source stays under zkbot/protos, generated outputs live in shared top-level language-specific directories or libraries following each language's conventions, Python supports both pb2/grpc and BetterProto outputs from the same source, and code generation entrypoints are unified across Python, Java, and Rust.

## Symptoms

- Venue integrations vendor their own zk/*_pb2.py trees under venue-local proto directories, for example venue-integrations/oanda/oanda/proto/zk and venue-integrations/ibkr/ibkr/proto/zk
- Oanda relies on a sys.path injection shim in venue-integrations/oanda/oanda/proto/__init__.py so absolute imports like from zk.rtmd.v1 import rtmd_pb2 resolve
- tools/trade_doctor.py generates temporary pb2/grpc stubs on demand instead of consuming one shared generated Python proto package
- scripts/gen_proto.sh currently generates BetterProto output into libs/zk-datamodel/src/zk_datamodel, while Rust and Java use separate generation paths, so there is no single unified codegen workflow

## Impact

This creates duplicated generated artifacts, inconsistent import behavior, repeated proto copies across venues, higher risk of schema drift between integrations, and unclear ownership of Python pb2 versus BetterProto outputs. It also makes packaging and migration harder because proto runtime code is not organized as a shared language-level library boundary.

## Proposed Options Or Fixes

1. Create explicit top-level language output areas: keep zkbot/protos as schema source only, then generate shared outputs into language-specific locations such as a Python proto library, existing Java generated layout, and Rust generated layout following each language's conventions
2. For Python, support both outputs from the same schema source: a shared pb2/grpc package for low-level adaptor and gRPC usage, and a shared BetterProto package for higher-level datamodel usage; do not keep venue-local generated proto copies
3. Unify code generation entrypoints behind one repo-level workflow or wrapper so Python, Java, and Rust proto generation are driven consistently, even if the underlying generators differ
4. Migrate venue integrations and tools to import the shared generated Python proto package, then delete vendored proto trees and path hacks from venue packages
5. Decide whether generated artifacts are committed or fully reproducible during build, and document the policy clearly for all three language targets

## Open Questions

- What top-level directory naming should be used for generated language outputs: python/, java/, rust/ or libs/<lang-specific-package> within existing layouts?
- Should Python pb2/grpc and BetterProto outputs live in separate packages or under one top-level python generated tree with clear subpackages?
- Do we want Buf to orchestrate all language outputs directly, or should a repo wrapper script call Buf plus Python-specific generators such as betterproto and grpc_tools?
- Should trade_doctor and similar tools consume only the shared generated Python package, or keep an on-demand fallback for isolated tool execution?
