# zb-00028 Unify protobuf code generation into shared per-language generated libraries

- Ticket: zb-00028
- Date: 2026-04-18
- Time: 13:44 +08
- Domain: design, gw, mdgw, engine, strategy
- Reporter: agent
- Status: resolved

## Description

The repo currently duplicates generated Python protobuf code inside venue packages, generates BetterProto models into a separate shared library, and uses separate ad hoc generation paths for Python, Java, and Rust. The refactor should be broader than proto codegen alone: schema source should stay under zkbot/protos, generated outputs should live in shared top-level language-specific directories or libraries following each language's conventions, Python should support both pb2/grpc and BetterProto outputs from the same source, code generation entrypoints should be unified across Python, Java, and Rust, and the surrounding repository structure should make active versus legacy code, dependency direction, and deployment-time build requirements explicit.

## Requirements

- Keep `zkbot/protos` as the sole schema source-of-truth.
- Separate top-level language roots clearly, with `python/`, `java/`, and `rust/` each treated as first-class areas.
- Preserve cross-cutting concern separation as well, so language-root separation and concern-level separation both exist at the same time.
- For Python, treat `zkbot/services/*` as reference-only legacy code except `zkbot/services/zk-refdata-svc`, and make that status explicit in the tree and in docs.
- For Java, normalize service layout into service-level modules such as `java/pilot-service` and future `java/pts-service`, with shared proto or utility modules clearly separated.
- Make inter-package and inter-language references explicit and consistent; imports should not rely mainly on local dev-stack assumptions such as cwd, PYTHONPATH injection, vendored generated files, or workspace-relative path hacks.
- For real deployment and CI/CD, specify dependency and build contracts clearly:
  - schema location
  - generated artifact ownership
  - committed vs build-time outputs
  - Python to proto dependencies
  - Rust to proto dependencies
  - Java to proto dependencies
  - allowed dependency direction between packages
- Treat this as a repository structure and dependency-contract refactor, not only a proto generation cleanup.

## Symptoms

- Venue integrations vendor their own zk/*_pb2.py trees under venue-local proto directories, for example venue-integrations/oanda/oanda/proto/zk and venue-integrations/ibkr/ibkr/proto/zk
- Oanda relies on a sys.path injection shim in venue-integrations/oanda/oanda/proto/__init__.py so absolute imports like from zk.rtmd.v1 import rtmd_pb2 resolve
- tools/trade_doctor.py generates temporary pb2/grpc stubs on demand instead of consuming one shared generated Python proto package
- scripts/gen_proto.sh currently generates BetterProto output into libs/zk-datamodel/src/zk_datamodel, while Rust and Java use separate generation paths, so there is no single unified codegen workflow
- The current tree mixes language boundaries and cross-cutting concern boundaries inconsistently, so generated code, runtime code, and legacy reference code are not clearly separated
- Python runtime ownership is ambiguous: most of zkbot/services is effectively reference-only except zk-refdata-svc, but the repo structure does not make that explicit
- Java service layout is not yet normalized into service-level modules such as java/pilot-service and future java/pts-service with clearly separated shared dependencies
- Many imports and references work mainly because of local dev-stack assumptions such as cwd, PYTHONPATH injection, generated file placement, or local workspace-relative paths rather than explicit package or build dependencies
- Deployment and CI dependency contracts are unclear, especially for cross-language references such as Python or Rust consuming generated schema artifacts, Java depending on shared proto outputs, and the exact source-of-truth path for schemas

## Impact

This creates duplicated generated artifacts, inconsistent import behavior, repeated proto copies across venues, higher risk of schema drift between integrations, and unclear ownership of Python pb2 versus BetterProto outputs. It also makes packaging, migration, CI, and deployment harder because proto runtime code is not organized as a shared language-level library boundary, active versus legacy code paths are not explicit, and inter-language or inter-package dependency rules are not cleanly specified outside the local dev environment.

## Proposed Options Or Fixes

1. Create explicit top-level language output areas: keep zkbot/protos as schema source only, then generate shared outputs into language-specific locations such as a Python proto library, existing Java generated layout, and Rust generated layout following each language's conventions
2. For Python, support both outputs from the same schema source: a shared pb2/grpc package for low-level adaptor and gRPC usage, and a shared BetterProto package for higher-level datamodel usage; do not keep venue-local generated proto copies
3. Unify code generation entrypoints behind one repo-level workflow or wrapper so Python, Java, and Rust proto generation are driven consistently, even if the underlying generators differ
4. Migrate venue integrations and tools to import the shared generated Python proto package, then delete vendored proto trees and path hacks from venue packages
5. Decide whether generated artifacts are committed or fully reproducible during build, and document the policy clearly for all three language targets
6. Introduce an explicit repository-boundary cleanup stage that defines active versus legacy code, especially for Python, where zkbot/services should be treated as reference-only except zk-refdata-svc
7. Normalize top-level language layout and cross-cutting concern layout together: language roots such as python/, java/, and rust/ should coexist with clear concern-level separation such as shared proto libs, services, libraries, and legacy reference code
8. Normalize Java into service-level modules such as java/pilot-service, java/pts-service, and shared dependencies like java/zk-proto-java instead of keeping an implicit monolith
9. Define and document allowed dependency directions so runtime packages no longer depend on dev-stack-only path behavior or vendored generated code
10. Add a deployment and CI contract matrix that states clearly what is source-of-truth, what is generated, what is committed, what is built at package-build time, and what each language runtime depends on
11. Make active versus legacy code ownership explicit in the repository layout so Python reference-only paths are clearly distinguished from active runtime paths
12. Use the refactor to clean up inconsistent inter-reference patterns across the repo, especially where current behavior only works under the local dev stack

## Open Questions

Resolved (see Resolved Direction below):
- Top-level directory naming: language-rooted `python/`, `java/`, `rust/` trees under `zkbot/`.
- Python outputs: two separate libs (`python/zk-proto-pb`, `python/zk-proto-betterproto`).
- Codegen driver: single repo wrapper `scripts/gen_proto.py` calling buf + Python generators.
- trade_doctor: consumes shared `zk-proto-pb` only, no on-demand fallback.

Still open (need decision before Stage 4+ work):
- How should the repo explicitly mark active production code versus reference-only legacy code under Python paths? Options: move inactive `zkbot/services/*` into `python/legacy/services/`, keep in place with a top-of-file marker, or add a `LEGACY.md` index per reference-only package.
- What dependency matrix should be documented for CI and deployment so that cross-language references are explicit and do not rely on local dev-stack behavior (PYTHONPATH assumptions, workspace-relative paths, vendored generated code)?
- How much repository cleanup should happen in the same ticket versus a follow-on structural cleanup ticket if the proto work lands first?

## Resolved Direction (2026-04-18)

- **Schema source**: unchanged at `zkbot/protos/`; remains the sole source-of-truth for all languages.
- **Top-level language roots**:
  - `python/zk-proto-pb/` — shared pb2/grpc (installable package)
  - `python/zk-proto-betterproto/` — shared BetterProto (replaces `libs/zk-datamodel/`)
  - `python/legacy/` — reference-only snapshots of vendored venue proto trees, pre-refactor `zk-datamodel`, and legacy flat `.proto` outputs (inactive services may also land here pending the open question above)
  - `java/zk-proto-java/` — shared generated Java Gradle subproject
  - `java/pilot-service/` — current monolithic `java/src` restructured as a service subproject; future `java/pts-service` (post-trade-service) follows the same pattern and depends on `:zk-proto-java`
  - `rust/crates/zk-proto-rs/` — already the shared Rust crate; unchanged
- **Repository-structure goal**: top-level language-root separation and cross-cutting concern separation must both be preserved. This work is not complete if the repo only gets new generated proto libraries but still leaves active vs legacy paths and service-module boundaries ambiguous.
- **Python output split**: two separate top-level libraries (pb2 and BetterProto), not one tree with subpackages. Independent installs for venue adaptors vs. datamodel consumers.
- **Codegen driver**: single repo wrapper `scripts/gen_proto.py` that calls `buf generate` (Rust, Java, cpp) plus `grpc_tools.protoc` and `betterproto` for the two Python outputs, with `scripts/gen_proto_legacy_compat.py` as a post-pass over BetterProto. `make gen` is the sole entrypoint. `scripts/gen_proto.sh` is deleted.
- **trade_doctor**: consumes shared `zk-proto-pb` only; on-demand stub generation and the `.trade_doctor_generated/` cache are removed.
- **Commit policy**: Python pb2/BetterProto and Rust generated sources are committed; Java generated sources are built by Gradle at build time (status quo).
- **Venue migration**: `oanda` and `ibkr` drop vendored `oanda/proto/zk/**` and `ibkr/proto/zk/**` trees (snapshotted to `python/legacy/`), switch to `from zk_proto_pb.zk.*` imports, and declare `zk-proto-pb` in pyproject deps. The sys.path shim is removed.
- **Python runtime ownership (pending open question)**: `zkbot/services/*` is reference-only except `zk-refdata-svc`. Exact marking mechanism to be decided before Stage 4.
- **CI and deployment contract requirement (pending detailed matrix)**: final design must state explicitly which package or module owns each generated artifact, what is committed, what is built at package-build time, and how Python, Java, and Rust consumers reference shared proto outputs outside the dev stack.
- **Staging**:
  - Stage 1 — Introduce shared Python libs and `gen_proto.py`; no consumer changes.
  - Stage 2 — Migrate venue integrations and `trade_doctor` to `zk-proto-pb`; delete vendored trees.
  - Stage 3 — Migrate 81 `zk_datamodel` consumers to `zk-proto-betterproto`; retire `libs/zk-datamodel`.
  - Stage 4 — Rewrite `buf.gen.yaml`; restructure `java/` into `pilot-service` + `zk-proto-java` Gradle subprojects; mark legacy Python services.
  - Stage 5 — Cleanup `scripts/gen_proto.sh`, legacy flat `.proto` files, dangling refs; write ADR in `docs/system-arch/arch_design_decisions.md` including the deployment/CI dependency matrix (resolves the remaining open question).
