# zb-00013 Pilot and UI need strategy-manifest sync plus schema-driven strategy and bot forms

- Ticket: zb-00013
- Date: 2026-03-28
- Time: 12:59 +08
- Domain: pilot, strategy, engine
- Reporter: agent
- Status: resolved

## Description

Pilot and the Pilot UI now have engine and per-strategy manifests under service-manifests/engine, but the control-plane workflow does not yet treat strategy manifests as first-class schema sources. Auto schema sync and restart flows still focus on top-level service manifests, strategy authoring/editing is not yet driven by the selected strategy config schema, and bot authoring/editing is not yet driven by the engine config schema. This leaves the new manifest model partially documented but not operationalized in Pilot/UI.

## Symptoms

- Schema registry sync and restart flows do not clearly include service-manifests/engine/strategies/* manifests and schemas.
- Strategy create/edit flows are not yet guaranteed to load and validate against the selected strategy_type_key schema.
- Bot create/edit flows are not yet guaranteed to load and validate against the engine service-kind config schema.
- Operators still need a schema-free/manual registration path for strategies even after schema-driven authoring is added.

## Impact

Pilot/UI cannot fully realize the new engine-plus-strategy manifest model, so config validation, authoring UX, and schema reload behavior will drift from the architecture docs. Operators may create invalid strategy or bot configs, and manifest updates under strategy folders may not take effect after sync or restart.

## Proposed Options Or Fixes

1. Extend Pilot schema/manifest discovery so auto sync and restart paths include service-manifests/engine/strategies/* manifests and their referenced schemas.
2. Add a strategy manifest registry path in Pilot that resolves strategy_type_key to its manifest/schema and exposes that contract to the UI.
3. Update strategy create/edit UX and backend validation so the selected strategy_type_key drives schema-based config forms and validation, while keeping an explicit schema-free registration path for advanced/manual use.
4. Update bot create/edit UX and backend validation so engine_instance provided_config is driven by service-manifests/engine/manifest.yaml and engine_config.schema.json.
5. Document reload and restart semantics for both engine service-kind schema changes and per-strategy manifest/schema changes so operators know when a sync is enough and when a restart is required.

## Open Questions

- Should strategy manifests be synced into the existing schema registry tables, or loaded through a dedicated strategy-manifest registry path?
- What exact API contract should the UI use to fetch schema-driven authoring metadata for strategies and bots?
- Should schema-free strategy registration be a separate endpoint/mode, or a toggle within the same create/edit workflow?

## Resolution

- Resolved At: 2026-03-28 13:36 +08
- Summary: Added strategy_type_key as first-class field on cfg.strategy_definition (V10 migration). Extended SchemaRegistrySyncer to discover nested strategy manifests under service-manifests/engine/strategies/ with folder-name validation. Added /v1/schema/strategies/{strategyTypeKey} and .../config API endpoints. Replaced raw JSON textareas with SchemaForm in StrategiesPage (strategy config) and OnboardBotPage (engine provided_config), with fallback to free-form JSON when no schema exists. BotService.startBot now sources ZK_STRATEGY_TYPE_KEY from strategy_definition.strategy_type_key instead of code_ref. Bot detail editing deferred to follow-up ticket.
