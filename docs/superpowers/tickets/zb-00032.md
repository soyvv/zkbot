# zb-00032 Enrich IBKR refdata universe resolver beyond explicit contract lists

- Ticket: zb-00032
- Date: 2026-04-21
- Time: 23:22 +08
- Domain: refdata, gw, design, pilot
- Reporter: agent
- Status: resolved

## Description

IBKR refdata loading currently treats config.universe as the full input universe and only qualifies each known contract spec through IBKR reqContractDetails. That is correct for qualification, but insufficient for operator workflows that need to derive a universe from a file, scanner, or other source. IBKR does not provide a bulk enumerate-all-symbols API, so the loader needs an explicit universe_source resolver layer before the existing qualification loop.

## Symptoms

- IbkrRefdataConfig only exposes universe: tuple[str, ...] for contract strings such as AAPL-USD-STK-SMART
- IbkrRefdataLoader.load_instruments returns [] immediately when universe is empty
- load_instruments iterates config.universe and calls ContractTranslator.qualify followed by ib.reqContractDetailsAsync, which qualifies known specs but does not discover candidates
- refdata_config.schema.json only documents a universe array and has no file/scanner/source-of-truth model
- load_market_sessions also depends on the same configured universe/cache, so sessions cannot refresh for an externally desired universe unless that universe was hand-specified
- IBKR discovery primitives are fragmented: reqMatchingSymbols is small typeahead, scanners return top-N filtered results, option discovery needs an underlying, and none provide all US ETFs or all tradable symbols

## Impact

Operators cannot onboard broad IBKR universes such as US ETFs, US equities, or liquid tradable sets from Pilot/refdata config without hand-entering contract strings. Empty universe silently produces no refdata, which is operationally valid but easy to misinterpret as successful discovery. Without a resolver layer, IBKR refdata refresh cannot be reproducibly driven from external symbol catalogs or IBKR scanner definitions.

## Proposed Options Or Fixes

1. Add a universe_source discriminated object to ibkr refdata config while preserving the existing explicit universe behavior for backwards compatibility
2. Implement _resolve_universe() before the current qualification loop: explicit uses config.universe, file reads CSV/JSON candidate symbols and maps them into IBKR contract specs, scanner invokes reqScannerSubscription/ScannerSubscription and returns top-N symbols, and later option-chain support can call reqSecDefOptParams for OPT universes
3. Keep reqContractDetails as the qualification/enrichment step after resolving candidates, because it attaches contract identifiers, minTick, multiplier, liquidHours, and filters invalid/delisted/wrong-exchange names
4. Reuse or extract the existing IBKR TokenBucketRateLimiter for refdata qualification so large universes are batched under IBKR pacing constraints
5. Add a reproducible dev path for broad ETFs/stocks: mount or commit a small sample CSV under devops/data/ for tests, and document production sources such as NASDAQ ETF lists, exchange traded-security files, CME catalogs, or vendor CSVs
6. Expose clear Pilot/operator config examples for explicit, file, and scanner source types, including conservative defaults for max rows, exchange, currency, secType, and rate limits

## Open Questions

- What should the canonical file schema be for file-based universe sources: raw symbols plus defaults, or fully specified IBKR contract strings?
- Should file paths be absolute only under the dependency-contract style, or should Pilot/refdata support named data artifacts mounted under a configured data root?
- Which scanner presets should be first-class examples: TOP_TRADE_COUNT_USD for STK.US.MAJOR, ETF-specific scanner filters, or user-supplied arbitrary scanner config?
- Should the loader fail fast when universe and universe_source are both empty, or keep the current empty-list behavior?
- Where should broad vendor datasets live operationally: devops/data for local dev, object storage for prod, or a future refdata artifact registry?

## Resolution

- Resolved At: 2026-04-29 00:26 +08
- Summary: Foundation shipped via zb-00033: UniverseResolver Protocol + universe_source discriminated config + ExplicitResolver + NasdaqSymbolDirectoryResolver are all in IbkrRefdataLoader. The architectural goal — replace 'config.universe is the full input' with 'pluggable universe_source resolves candidates before qualification' — is achieved. Loader's _resolve_universe() dispatches to the right resolver, then the existing reqContractDetails qualification loop runs unchanged. Operators now compose universes from explicit lists or the Nasdaq Trader symbol directory; ETF vs STOCK proto-typed via zb-00039 dict items. Additional resolver implementations (file CSV/JSON, scanner reqScannerSubscription, option chain reqSecDefOptParams) listed in option 2 are scoped as separate tickets when each becomes operationally needed — adding a new resolver is a one-file change against the established UniverseResolver Protocol. 195 IBKR tests green.
