# zb-00036 Refdata-svc registry registration gated on initial refresh; Pilot proxy unavailable for ~50min

- Ticket: zb-00036
- Date: 2026-04-27
- Time: 23:30 +08
- Domain: refdata, pilot
- Reporter: agent
- Status: resolved

## Description

main.py registers the service in NATS KV (svc.refdata.<id>) only after coordinator.refresh_all_periodic(enabled) completes. With the IBKR universe now ~11k contracts and qualification at ~2/sec, this blocks Pilot's discovery for ~50 minutes after startup. During that window, POST /v1/refdata/venues/{venue}/refresh returns 503 'no refdata service registered'.

## Symptoms

- POST /v1/refdata/venues/oanda/refresh returned 503 immediately after refdata-svc was running and bootstrap-registered, because the initial IBKR refresh (~50min) hadn't yet completed.
- Pilot's DiscoveryResolver.findRefdata returns null until ServiceRegistry.start() runs.
- Manual operators can't trigger refresh on any venue, including the small ones (OANDA <1s, OKX <1s), until IBKR finishes.

## Impact

Operators lose all refresh-via-UI capability for the first ~50 minutes after each restart of zk-refdata-svc. The new TriggerVenueRefresh RPC works directly on the gRPC port but Pilot can't proxy it. UX implication: 'why is the refresh button broken' until run completes.

## Proposed Options Or Fixes

1. Register in discovery as soon as gRPC is listening (move ServiceRegistry.start() before initial refresh). Add a 'data_ready' state field to the registration so consumers can decide whether to query it for live data — a separate concern from 'is the RPC port reachable'.
2. Or: kick off coordinator.refresh_all_periodic() as a background task (not awaited) and register immediately. Subsequent reads return the previous-run state until the new run lands.
3. Or: keep the gating but emit a coarse 'startup_progress' KV value so Pilot can show 'refdata starting (3 of 4 venues)' instead of 503.

## Open Questions

- Is 'service registered' meant to imply 'has data' or 'RPC reachable'? Today it means both, which is the source of the gating. The new TriggerVenueRefresh RPC inverts the assumption — it works without prior data.

## Resolution

- Resolved At: 2026-04-29 00:22 +08
- Summary: Decoupled NATS KV service registration from initial refresh completion in main.py. Service now registers as soon as the gRPC server is listening; initial refresh runs as a background task (asyncio.create_task) that flows directly into the periodic schedule. Pilot's DiscoveryResolver sees svc.refdata.<id> within seconds of refdata-svc startup, not after the ~50min IBKR initial refresh. Manual TriggerVenueRefresh + Pilot proxy work immediately. Consumers needing 'is the current dataset fresh' semantics can poll cfg.refdata_refresh_run via the existing GetRefreshRun RPC. 113 refdata-svc tests green.
