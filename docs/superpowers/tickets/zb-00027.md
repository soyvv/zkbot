# zb-00027 KV heartbeat timeout is misclassified as fencing after NATS reconnect

- Ticket: zb-00027
- Date: 2026-04-16
- Time: 20:48 +08
- Domain: oms, gw, design
- Reporter: codex
- Status: resolved

## Description

The shared KV heartbeat loop currently treats any JetStream KV update error as a fencing event. After a transient NATS disconnect/reconnect, a heartbeat update can fail with a timeout and the service shuts down as if another instance had taken ownership, even when there is no evidence of a competing writer.

## Symptoms

- Logs show async_nats disconnect/reconnect immediately followed by 'KV heartbeat CAS failed — fenced, stopping' with error=timed out
- OMS logs 'KV fencing detected — another instance owns this identity' and exits even though the underlying error is a timeout, not an explicit CAS conflict
- Operational restarts or brief broker interruptions can cause OMS or other registered services to self-terminate and rely on later manual recovery

## Impact

This creates false-positive split-brain shutdowns, reduces runtime availability during ordinary NATS recovery events, and makes the control-plane ownership story misleading because timeout-class transport failures are reported as confirmed ownership loss.

## Proposed Options Or Fixes

1. Classify explicit CAS-conflict or wrong-revision errors as true fencing, but treat timeout/disconnect/reconnect errors as retryable heartbeat failures
2. Add bounded retry with backoff for retryable KV update failures and only transition to a non-fenced degraded or unavailable state after repeated failures
3. Introduce a distinct shutdown/error reason such as RegistryUnavailable or HeartbeatFailed so logs and service behavior clearly distinguish ownership loss from transport instability
4. Add integration coverage that simulates NATS disconnect/reconnect during heartbeats and verifies services do not self-fence on timeout-only failures

## Open Questions

- What concrete async_nats or JetStream error variants/messages should be considered authoritative CAS conflicts versus retryable transport errors?
- Should services keep serving while registry heartbeats are temporarily degraded, or should they enter a separate non-discoverable drain mode after a threshold?
- Do we want the retry/failure policy centralized entirely in zk-infra-rs so OMS, GW, engine, and mdgw all inherit the same semantics?

## Resolution

- Resolved At: 2026-04-16 21:02 +08
- Summary: Classified heartbeat update errors by kind: WrongLastRevision (CAS conflict) triggers immediate fencing; TimedOut and other transport errors retry up to 5 consecutive times before signaling RegistryUnavailable. Added HeartbeatLost shutdown reason distinct from Fenced so logs and caller behavior correctly distinguish ownership loss from transport instability. All 5 dependent services (OMS, GW, RTMD-GW, engine) compile clean and inherit the fix via zk-infra-rs. 66 unit tests + 5 doc-tests pass.
