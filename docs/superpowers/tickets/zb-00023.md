# zb-00023 OMS should import gateway-only open orders during reconciliation

- Ticket: zb-00023
- Date: 2026-04-13
- Time: 23:30 +08
- Domain: oms
- Reporter: codex
- Status: open

## Description

OMS order reconciliation currently only converges OMS-known open orders against gateway state. Open orders that exist on the gateway but have no corresponding OMS order (e.g. orders placed via venue UI, orders from a prior OMS instance that didn't persist, or orders placed by another system sharing the same venue account) are silently ignored.

This ticket tracks the design and implementation of a safe import path for gateway-only open orders into OMS.

## Context

An initial attempt to import gateway-only orders was made in zb-00021 using `identify_gw_only_orders()`. That implementation was reverted because the synthetic `OrderReport` it produced carried only state-level fields (status, filled_qty, unfilled_qty, avg_price) with `order_id=0`. The resulting OMS order had `instrument_id=0`, `qty=0.0`, `price=0.0`, no route account, and no side/order-type — making it a malformed placeholder rather than a usable order.

## Requirements

Before implementing, the following contract must be defined:

1. **Source fields from ExchOrder**: What fields does ExchOrder reliably carry? At minimum: `order_ref`, `instrument_code`, `side`, `order_type`, `qty`/`unfilled_qty`, `price`, `filled_qty`, `avg_price`, `status`. Verify which gateways populate these.
2. **Account resolution**: ExchOrder has no `account_id`. How to resolve — default account from confdata, or require the GW to report it?
3. **Instrument resolution**: How to map `instrument_code` from ExchOrder to OMS `instrument_id`. Must handle refdata lookup failures gracefully.
4. **Order economics**: qty, price, side, order_type must all be populated for the imported order to be meaningful for position/balance calculations and UI display.
5. **Route account**: The imported order needs a route account symbol for proper executor routing.
6. **Timestamps**: What timestamps to use (GW report time? order creation time from ExchOrder?).
7. **UI/downstream treatment**: Should imported orders be visually distinguished as "externally discovered"? Should they be editable/cancellable from OMS?

## Proposed Approach

- Extend `ExchOrder` or the GW query response to carry the required fields (instrument, side, qty, price, order_type)
- Build a proper `OrderReport` with `ExecReport` (not just `OrderStateReport`) that carries full order shape
- Use `OrderSourceNonTq` to mark as externally originated
- Add the import step to all three reconcile paths (startup, late-discovery, periodic) via the shared `reconcile_gw_orders` helper
- Add integration tests that verify imported orders have complete economics
