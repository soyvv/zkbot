# zb-00006 Wire live spot exposure projection and remove mixed balance/position dependence

- Ticket: zb-00006
- Date: 2026-03-25
- Time: 23:55 +08
- Domain: oms, gateway, spot, backtest
- Reporter: agent
- Status: open

## Description

The refactor design says spot asset balances remain canonical `Balance`, but OMS may derive
sellable spot exposure as an operational `Position` projection.

Utilities for this projection already exist, and backtest init uses them. Live OMS does not
yet consistently apply the same spot-exposure projection at bootstrap/reconcile points.
Separately, the gateway/OMS boundary still uses one mixed `BalanceUpdate` carrier for both
asset balances and position-like exposure.

Need to finish the live spot projection path and prepare the runtime for separate balance
and position update flows.

## Symptoms

- live OMS spot updates mostly populate balances only
- spot exposure derivation utilities are not wired into live bootstrap/reconcile
- gateway semantic pipeline still publishes balances and positions through the same mixed message
- replay/backtest and live OMS are not using the same spot-exposure architecture

## Impact

- spot sell-side operational exposure is inconsistent between live and replay
- balance vs position ownership boundaries remain blurry at the gateway ingress boundary
- future venue integrations will keep reintroducing the same ambiguity

## Proposed Options Or Fixes

1. Use `derive_spot_positions_from_balances` in live OMS bootstrap/reconcile flows.
2. Define precedence between derived spot exposure and explicit exchange position snapshots.
3. Add a distinct gateway-originated position update path and keep balance updates asset-scoped.
4. Reduce OMS reliance on classifying one mixed gateway payload into spot-vs-nonspot branches.
5. Update replay/backtest to validate the same spot projection semantics as live OMS.

## Acceptance Criteria

- spot balances remain canonical balances
- live OMS exposes derived spot executable exposure where required for operations
- gateway/runtime paths are moving toward separate balance and position update semantics
- replay covers the same spot projection rules as live OMS
