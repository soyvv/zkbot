# zb-00009 Guarantee immediate OMS convergence for gateway synthetic rejects

- Ticket: zb-00009
- Date: 2026-03-26
- Time: 00:15 +08
- Domain: oms, orders, runtime, gateway
- Reporter: agent
- Status: open

## Description

An OKX rejected order remained `Pending` in OMS until the background reconcile job
scanned and repaired it.

This indicates the immediate reject-handling path is not reliable enough in the
live service path, even if later reconcile can repair state eventually.

Need to guarantee that a gateway synthetic reject transitions OMS order state
immediately, without depending on order recheck or restart reconcile.

## Symptoms

- gateway rejects a place-order request
- OMS receives or should receive the synthetic reject
- order still appears `Pending` until later reconcile

## Impact

- live OMS state is temporarily wrong
- order-status consumers cannot trust immediate execution feedback
- reconcile is doing correctness work that should have happened in the hot path

## Proposed Options Or Fixes

1. Verify the current OMS V2 reject path handles both `error_info` and
   `rejection_info` for synthetic gateway rejects.
2. Add end-to-end service tests for adapter reject -> gateway synthetic report ->
   OMS `Rejected`.
3. Cover empty `exch_order_ref` plus valid `order_id` resolution explicitly.
4. Audit publish/persist ordering to ensure the rejected state reaches snapshot,
   event, and Redis paths immediately.
5. Keep reconcile as a safety net only, not the primary reject path.

## Acceptance Criteria

- a gateway synthetic reject moves the order out of `Pending` immediately
- OMS snapshot, persisted state, and published order update all reflect `Rejected`
- the same order does not need background reconcile to become correct
